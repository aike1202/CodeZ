## 一、P0 — Agent Types 声明

### 是什么

真实 Claude Code 在系统提示词中声明了 6 种子 Agent 类型，每种有不同的工具权限和使用场景：

```
Available agent types for the Agent tool:
- claude: 通用任务，拥有全部工具
- Explore: 只读搜索，无 Edit/Write/NotebookEdit 权限
- Plan: 架构设计，只读，用 EnterPlanMode/ExitPlanMode
- general-purpose: 复杂多步任务，全部工具
- claude-code-guide: 回答 CLI/API 问题，仅 Glob/Grep/Read/WebFetch/WebSearch
- statusline-setup: 状态栏配置，仅 Read/Edit
```

### 为什么需要

模型在任务中需要知道"可以 spawn 什么类型的子 Agent"。没有这个声明，模型不会主动使用 Agent 工具来做并行探索或任务委派。

### 实现内容

| 层 | 做什么 |
|----|--------|
| **System Prompt** | 在当前 `<pending_features>` 处替换为完整的 Agent types 声明 |
| **ToolManager** | 新增 `getToolsByAgentType(type)` 方法，按类型过滤工具列表 |
| **AgentRunner** | 支持 `roleConfig` 参数，spawn 子 Agent 时按角色限制工具 |

### 在 System Prompt 中的样子

```
<available_agent_types>
When using the Agent tool, specify a subagent_type to select a specialized
agent. Each type has different tool permissions:

- general-purpose: Catch-all for any task. All tools available.
- Explore: Read-only search agent. No Edit/Write/NotebookEdit/Bash/PowerShell.
- Plan: Architecture design agent. Read-only, no file modifications.

When launching multiple agents for independent work, send them in a single
message with multiple tool uses so they run concurrently.
</available_agent_types>
```

**工作量**：小（~100 行，SystemPromptService + ToolManager 各改一点）

## 二、P1 — Plan 模式（EnterPlanMode / ExitPlanMode）

### 是什么

Plan 模式是一个"先设计、后执行"的工作流。模型在进入 Plan 模式后：
- **只能使用只读工具**（Read、Glob、Grep、ListFiles）
- 探索代码库、分析需求
- 输出设计计划给用户审批
- 用户批准后退出 Plan 模式，恢复全部工具

| 文件 | 变更 |

## 四、P2 — Worktree 隔离（EnterWorktree / ExitWorktree）

### 是什么

Git Worktree 允许多个并行工作目录共享同一个 `.git` 仓库。当需要：
- 多个子 Agent 同时修改不同文件
- 隔离测试某个改动不影响主工作区
- 安全地做大规模重构

Worktree 提供了文件系统级别的隔离。

### 后端（Main Process）

**新增文件**：

| 文件 | 职责 |
|------|------|
| `src/main/tools/builtin/EnterWorktreeTool.ts` | 创建临时 git worktree，切换到新分支 |
| `src/main/tools/builtin/ExitWorktreeTool.ts` | 清理 worktree 目录，合并/丢弃分支 |
| `src/main/services/WorktreeService.ts` | 封装 git worktree 命令 |

**EnterWorktreeTool 行为**：
```
模型调用 EnterWorktree({name: "fix-login"})
  → WorktreeService.create("fix-login")
    → git worktree add .claude/worktrees/fix-login -b fix/fix-login
    → 返回 { path: "/abs/path/.claude/worktrees/fix-login", branch: "fix/fix-login" }
  → AgentRunner 切换 cwd 到 worktree 目录
  → Inject "Now working in isolated worktree fix-login" system message
```

**ExitWorktreeTool 行为**：
```
模型调用 ExitWorktree({action: "keep"})
  → 保留 worktree 目录和分支，不删除
  → AgentRunner 恢复 cwd 到原始 workspace
  → 通知用户 "Worktree fix-login kept at <path>"

ExitWorktree({action: "remove", discardChanges: true})
  → git worktree remove .claude/worktrees/fix-login --force
  → git branch -D fix/fix-login
  → 清理目录
```

**安全性**：
- worktree 只能创建在 `.claude/worktrees/` 下（sandbox）
- 不能访问 workspace 之外的路径
- 原始 workspace 的 `.gitignore` 仍然生效

### 前端（Renderer）

**修改文件**：

| 文件 | 变更 |
|------|------|
| `src/renderer/src/stores/workspaceStore.ts` | 新增 `activeWorktree: { name, path } | null` |
| `src/renderer/src/components/TopBar.tsx` | 显示当前 worktree 名称（如 "🔀 fix-login"） |

**TopBar 显示示例**：
```
CodeZ  🔀 fix-login    [main]    Opus 4.8
```

**工作量**：大（~500 行，2 工具 + WorktreeService + git 命令封装 + workspaceStore 扩展 + TopBar 改动）

## 五、P3 — 远期规划（详细描述）

### 5.1 Monitor / Cron / ScheduleWakeup

#### 是什么

真实 CC 有三层调度能力：

| 工具 | 用途 | 示例 |
|------|------|------|
| **Monitor** | 实时监听外部事件（文件变更、日志、WebSocket） | "每次 dev.log 出现 ERROR 就通知我" |
| **CronCreate** | 定时任务（重复或一次性） | "每 5 分钟检查一次 CI 状态" |
| **ScheduleWakeup** | Agent 自调度的休眠-唤醒 | Agent 说"1 小时后提醒我检查部署" |

#### 后端设计

**Monitor**：
```
Monitor({command: "tail -f /var/log/app.log | grep --line-buffered ERROR"})
  → 后台启动子进程 tail -f
  → 每行 stdout 作为事件推送到前端
  → 用户可以随时取消
```

**Cron**：
```
CronCreate({cron: "*/5 * * * *", prompt: "检查 npm test 是否通过"})
  → Node.js setTimeout 或 node-cron 调度
  → 到时自动 enqueue prompt
  → 7 天自动过期
```

**ScheduleWakeup**：
```
ScheduleWakeup({delaySeconds: 3600, prompt: "继续上次的部署任务"})
  → Agent 进入空闲状态
  → delaySeconds 后自动发送 prompt
  → 结合 prompt caching（5min TTL）选择最佳延迟
```

#### 前端

| 组件 | 职责 |
|------|------|
| `ScheduledTasksPanel.tsx` | 显示活跃的 Cron/Monitor 列表 |
| `CronStatusBadge.tsx` | TopBar 上的定时任务数量角标 |

#### 存储

```
~/.codez/scheduled_tasks.json  ← Monitor 和 Cron 的持久化
```

**工作量**：中（~700 行，4 工具 + 2 前端组件 + 调度引擎 + 持久化）

### 5.2 Workflow 编排

#### 是什么

Workflow 是真实 CC 的多 Agent 编排引擎。核心概念：

```
用户任务
  → Workflow 脚本（JavaScript）
    → pipeline(items, stage1, stage2, ...)  ← 流水线，无 barrier
    → parallel(thunks)                       ← 并行，有 barrier
    → agent(prompt, opts)                    ← 启动子 Agent
    → phase(title)                           ← 进度分组
  → 多个子 Agent 按拓扑执行
  → 结果汇总、去重、验证、合成
```

#### 核心原语

| 原语 | 行为 |
|------|------|
| `pipeline(items, stage1, stage2)` | 每个 item 流经所有 stage，item 之间无 barrier |
| `parallel(thunks)` | 所有 thunk 并发执行，等待全部完成（barrier） |
| `agent(prompt, {schema, model, isolation})` | spawn 一个子 Agent，可返回结构化数据 |
| `phase(title)` | 进度分组标签 |

#### 为什么不是 P0/P1

Workflow 会放大当前系统的不足：
- 工具选择混乱 → 多个 Agent 各自选错工具
- 文件写入冲突 → 并行修改同一个文件
- 权限不完整 → 子 Agent 绕过权限
- 验证闭环不稳定 → 多个 Agent 各自失败

必须等单 Agent 闭环完全稳定后再做。

#### 后端设计

**新增模块**：

| 模块 | 职责 |
|------|------|
| `src/main/agent/WorkflowEngine.ts` | 解析 Workflow 脚本，调度 pipeline/parallel/agent |
| `src/main/agent/WorkflowScriptParser.ts` | 安全沙箱执行用户 JS 脚本 |
| `src/main/agent/AgentBlackboard.ts` | 跨 Agent 共享上下文（API 契约、关键发现、文件边界） |

**Workflow 工具**：
```ts
Workflow({
  script: `
    export const meta = { name: 'review-changes', ... }
    const results = await pipeline(DIMENSIONS,
      d => agent(d.prompt, {schema: FINDINGS_SCHEMA}),
      review => parallel(review.findings.map(f => 
        () => agent("Verify: " + f.title, {schema: VERDICT_SCHEMA})
      ))
    )
  `
})
```

**安全性**：
- 脚本在受限沙箱中执行（无 fs/net 访问）
- `Date.now()` / `Math.random()` 被禁止（保证幂等 resume）
- 最大 1000 agent 调用上限
- 并发上限 min(16, cpuCores - 2)

#### 前端

| 组件 | 职责 |
|------|------|
| `WorkflowProgressPanel.tsx` | 多轨道 Agent 进度展示 |
| `AgentTrack.tsx` | 单个 Agent 的独立进度条 |
| `BlackboardView.tsx` | 共享上下文可视化 |

**工作量**：大（~2000 行，Workflow 引擎 + 脚本解析 + Blackboard + 3 前端组件 + 安全沙箱）

### 5.3 MCP 接入

#### 是什么

MCP（Model Context Protocol）是把外部系统接入 Agent 的标准协议。一个 MCP Server 可以提供：

| 能力 | 示例 |
|------|------|
| **tools** | 查询数据库、控制浏览器、读 GitHub issue |
| **resources** | 文档 schema、API 描述、设计稿 |
| **prompts** | 可复用提示模板 |

#### 为什么不是 P0/P1

MCP Server 接入后，Agent 会获得更多工具。但如果 Agent 本身的基础执行能力（读文件、搜文件、改文件、跑验证）不稳固：
- 外部工具会分散模型注意力
- 增加权限管理复杂度
- 错误更难排查

#### MCP 调用流程

```
Agent 请求调用 MCP 工具
  → Runtime 检查权限和参数
  → MCP Client 发送请求到 MCP Server（通过 stdio/SSE/WebSocket）
  → MCP Server 访问外部系统（数据库、浏览器、GitHub API）
  → 返回标准化响应
  → Agent 继续推理
```

#### 后端设计

**新增模块**：

| 模块 | 职责 |
|------|------|
| `src/main/mcp/McpClient.ts` | MCP 协议客户端（stdio/SSE 传输） |
| `src/main/mcp/McpServerManager.ts` | 管理已注册的 MCP Server 生命周期 |
| `src/main/mcp/McpToolProxy.ts` | 把 MCP tools 代理为 CodeZ Tool 接口 |
| `src/main/tools/builtin/McpTool.ts` | 暴露给模型的统一 MCP 调用工具 |

**MCP Server 配置**（`~/.codez/mcp-servers.json`）：

```json
{
  "servers": [
    {
      "name": "browser",
      "command": "node",
      "args": ["mcp-servers/browser/server.js"],
      "env": {}
    },
    {
      "name": "github",
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": { "GITHUB_TOKEN": "${env:GITHUB_TOKEN}" }
    }
  ]
}
```

#### 前端

| 组件 | 职责 |
|------|------|
| `McpSettingsTab.tsx` | MCP Server 管理界面（安装/启用/禁用） |
| `McpServerStatus.tsx` | 每个 Server 的连接状态指示灯 |

#### 权限

MCP 调用走同一套权限系统，不能绕过。

**工作量**：大（~1500 行，McpClient + ServerManager + ToolProxy + 配置系统 + 2 前端组件）

### 5.4 Swarm 多 Agent 并发

#### 是什么

Swarm 是比 Workflow 更高层的多 Agent 协作框架。角色分工：

```
User
  → Manager Agent (需求理解、任务拆解、DAG、结果合并)
    → Scout Agent ×N (并行搜索、读取、总结上下文)
    → Coder Agent ×N (并行实现独立模块)
    → QA Agent (运行测试、验证、诊断失败)
  → AgentBlackboard (共享关键上下文)
  → Final Report
```

#### 角色权限

| 角色 | 职责 | 工具权限 |
|------|------|---------|
| Manager | 需求理解、任务拆解、DAG、合并 | search/read only，不写文件 |
| Scout | 快速搜索、读取、总结 | search/read only |
| Coder | 修改授权范围内文件 | search/read/Edit/Write、有限 shell |
| QA | 测试、验证、诊断 | read/shell，默认不写文件 |

#### 核心模块

| 模块 | 职责 |
|------|------|
| `RoleConfig` | 定义每种角色的 system prompt、工具列表、fileScope、并发数 |
| `SwarmDispatcher` | 接收 DAG → 拓扑排序 → 控制并发 → 启动 AgentRunner → 收集结果 |
| `AgentBlackboard` | 保存 API 契约、关键发现、文件责任边界、GoalSnapshot、DecisionLog |

#### 冲突检测

- 同一文件不能被多个 Coder 同时写入
- DAG 有环时校验失败
- 任一子 Agent 失败不破坏全局

#### 前端

| 组件 | 职责 |
|------|------|
| `SwarmDashboard.tsx` | 多 Agent 进度可视化 |
| `RoleLegend.tsx` | 颜色编码的角色图例 |
| `BlackboardPanel.tsx` | 共享上下文浏览器 |

**工作量**：大（~3000 行，SwarmDispatcher + RoleConfig + Blackboard + SwarmDashboard + 冲突检测 + DAG 校验）
