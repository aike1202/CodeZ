# 子智能体中断权威判定设计

## 背景

当前前端在加载或重新选择会话时，只要持久化记录中的子智能体仍为
`running`，就直接把它改成 `interrupted`，同时向用户输入框预填恢复提示。
这个判断没有查询主进程中的真实运行状态，因此用户切换会话后，即使父
Runner 和子智能体仍在后台正常执行，也会看到“已中断，可继续”。

中断状态目前还分散在三套不一致的模型中：后台句柄使用 `cancelled`，子智能体
结果只支持 `completed | failed`，前端记录额外使用 `interrupted`。这使取消信号、
工具结果和界面展示无法形成一致的状态闭环。

## 目标

- 只有主进程确认执行已不存在时，才能把持久化的 `running` 判定为中断。
- 正在后台执行的会话在切换离开并返回后仍显示真实运行状态。
- 子智能体真实中断时，像普通工具调用失败一样向主智能体返回
  `EXECUTION_INTERRUPTED`，由主智能体自行换方法继续完成原要求。
- 中断恢复不向用户输入框填写内容，也不要求用户发送“继续”。
- 用户主动停止整个任务时，同时停止父子执行，但不自动重新启动任务。
- 只有明确完成的执行才显示“已完成”。

## 非目标

- 不恢复子智能体内部逐 token 的 Provider 流。
- 不引入跨进程心跳、租约或分布式任务调度。
- 不修改普通失败、权限拒绝和 Provider 重试的现有交互。
- 不让用户在多个中断子智能体之间手动选择恢复项。

## 状态模型

统一子智能体终态语义：

| 状态 | 含义 | 主智能体行为 | 用户界面 |
| --- | --- | --- | --- |
| `running` | 主进程仍持有该执行 | 等待工具结果 | 显示运行中 |
| `completed` | 子智能体提交了有效结果 | 消费成功结果 | 显示已完成 |
| `failed` | 子智能体正常结束但任务失败 | 消费工具错误并继续决策 | 显示失败 |
| `interrupted` | 执行在没有正常终态的情况下消失或被子级取消 | 消费 `EXECUTION_INTERRUPTED` 并继续决策 | 显示已中断 |

`interrupted` 与用户主动停止父任务不同。用户主动停止会把父 Runner 和所有子执行
一起取消，整个主循环结束，不触发自动续跑。

## 权威运行态

主进程维护会话级运行注册信息，并通过只读 IPC 返回：

```ts
interface SessionRuntimeStatus {
  sessionId: string
  mainRunnerActive: boolean
  activeSubAgentIds: string[]
}
```

主 Runner 注册时记录其 `sessionId`，在完成、失败或停止的所有出口删除注册。
子智能体句柄记录 `sessionId` 和前端使用的 `subAgentId`，从而能够精确返回活跃子项。

前端选择会话时同时读取持久化会话和权威运行态：

1. `mainRunnerActive === true` 时，不修复任何 `running` 状态；优先保留内存中持续被
   流事件更新的消息，避免磁盘快照覆盖实时状态。
2. `mainRunnerActive === false` 且某条持久化记录仍为 `running` 时，才把该记录判定为
   `interrupted`。
3. 应用启动时不再批量把所有 `running` 直接改为中断；修复推迟到会话被选择并完成
   权威查询后执行。

这套判断不使用超时。长时间没有输出不等价于中断。

## 中断回传

### 父 Runner 仍在运行

父 Runner 的取消信号传入 `SubAgentManager.spawn()`。子智能体自身异常终止或被子级
取消时，`SubAgentRunner` 返回：

```json
{
  "ok": false,
  "error": {
    "code": "EXECUTION_INTERRUPTED",
    "message": "SubAgent execution ended before completion."
  },
  "data": {
    "status": "interrupted"
  }
}
```

该结果通过现有工具结果记录进入主智能体上下文。父 AgentRunner 保持运行，并在下一轮
由模型决定直接调查、重新委派或采用其他工具。前端不创建恢复提示。

### 应用崩溃或进程重启

`SessionRuntimeCoordinator.recoverSession()` 继续负责为没有 durable terminal event 的
工具调用写入中断结果。会话选择确认存在新修复的中断后，前端自动发起一次隐藏的内部
续跑请求：

- 不创建用户消息气泡。
- 不修改输入框内容。
- 不重复提交原始用户要求。
- 主智能体从账本中读取已经写入的 `EXECUTION_INTERRUPTED` 工具结果，并继续已有目标。
- 每个修复事件最多触发一次；保存 `interrupted` 后再次选择会话不会重复续跑。

内部续跑仍创建正常的主智能体回复消息，用户可以看到后续执行过程和最终答复。

### 用户主动停止

停止入口先中止父 Runner，再把同一父信号传播到其子智能体。该原因标记为
`USER_ABORTED`，不是可自动恢复的子工具中断。前端结束流并保存终态，不发起内部续跑。

## UI 规则

- `ExecutionLog` 接收消息的 `interrupted` 状态，不能只根据 `streaming` 推导终态。
- 任一已确认中断的子智能体存在时，顶部显示中断图标和“执行已中断”，不得显示绿色
  对勾或“已完成”。
- 子智能体卡片只在权威判定后显示“已中断”。
- 删除“已中断，可继续”中的用户操作暗示，改为“已中断”；后续处理由主智能体完成。
- 删除会话恢复对 `pendingPrompt` 的写入。

## 竞态处理

- 会话选择继续使用递增序号，丢弃过期 IPC 响应。
- 查询期间若主 Runner 完成，流回调会先更新内存会话；选择逻辑不得用较旧磁盘消息覆盖
  当前内存中的同一会话。
- 停止和正常完成都必须在 `finally` 路径清理主 Runner 与子智能体注册，防止幽灵运行态。
- 自动内部续跑只基于本次从 `running` 转换到 `interrupted` 的 `changed` 结果触发。

## 测试

新增或扩展测试覆盖：

1. 会话有活跃主 Runner 时，持久化的 `running` 子智能体保持运行，不生成输入提示。
2. 没有活跃 Runner 时，残留 `running` 被转换为 `interrupted`。
3. 会话恢复不会设置 `pendingPrompt`。
4. 真正中断生成 `EXECUTION_INTERRUPTED` 工具结果，父 AgentRunner 继续下一轮。
5. 用户主动停止会取消子智能体，但不会触发内部续跑。
6. 多个中断子智能体一次性通知主智能体，不遗漏且不重复。
7. `ExecutionLog` 对中断消息不显示“已完成”。
8. 现有子智能体协议、恢复、会话持久化和类型检查保持通过。

## 验收标准

- 复现截图场景时，只要后台 Runner 仍存在，子智能体保持“运行中”，输入框保持原样。
- 主进程不存在对应执行时，界面准确显示中断，主智能体自动继续处理，用户无需操作。
- 点击停止后没有继续执行的子工具，也不会自动重新启动主智能体。
- 页面上不存在同一执行同时显示“已完成”和“已中断”的状态组合。
