# Read 批量文件读取设计

## 目标

将现有 `Read` 工具升级为一次读取 1 至 8 个文件的并行工具，减少模型为了获取已知文件集合而产生的不必要 Agent Loop。

## 工具契约

`Read` 只接受必填 `files` 数组，不再兼容顶层 `file_path`、`offset`、`limit` 或 `pages`。

```json
{
  "files": [
    { "file_path": "src/App.tsx" },
    { "file_path": "src/db.ts", "offset": 100, "limit": 80 }
  ]
}
```

- `files` 最少 1 项，最多 8 项。
- 每项必须包含 `file_path`。
- 每项可独立包含 `offset`、`limit` 和保留字段 `pages`。
- 即使只读取一个文件，也必须使用长度为 1 的数组。

## 执行语义

现有单文件逻辑抽取为类内私有方法，完整保留以下行为：

- 工作区路径限制。
- 文件、目录和大小校验。
- 二进制检测。
- SHA256 与会话读取指纹。
- 默认完整读取去重。
- `offset`/`limit` 范围读取绕过去重。
- Notebook 渲染。
- 行号、单文件字符与字节截断规则。

顶层 `execute` 校验数组后使用 `Promise.all` 同时执行每个文件读取。批量读取不增加新的总字符或总字节限制，每个文件只受现有单文件规则约束。

## 输出结构

每个结果使用明确文件块分隔，并保持输入数组顺序：

```text
<file path="src/App.tsx">
1\t...
SHA256: ...
</file>

<file path="src/missing.ts">
Error: File not found.
</file>
```

单个文件失败不使整批调用失败，其他文件继续返回。参数结构错误或 `files` 超过 8 项时，整个调用返回顶层 `Error:`。

## 权限与兼容迁移

- `ReadTool.parameters_schema` 只暴露 `files`。
- 子 Agent 路径权限检查必须逐项检查 `files[].file_path`。
- 当前代码和测试中直接调用 `ReadTool` 的位置全部迁移为 `files` 数组。
- 不修改历史设计文档。
- 不读取、复用或修改废弃的 `ProjectAnalysisService.readManyFiles()`。

## 日志体验

- 长度为 1 的 `files` 按现有单文件读取行显示。
- 长度大于 1 时生成“并行读取 N 个文件”卡片。
- 卡片执行中和完成后保持展开，可手动收起。
- 文件子项保留点击预览能力。
- 卡片总耗时为该单次 `Read` 工具调用的起止时间。
- 部分文件失败不会隐藏其他文件项。

## 验证

- 单文件数组保持现有正文、SHA、指纹和范围读取行为。
- 2 至 8 个文件同时读取并保持输入顺序。
- 第 9 个文件触发顶层参数错误。
- 文件缺失、二进制或越界时其他文件仍返回。
- 顶层旧 `file_path` 参数被拒绝。
- 权限检查覆盖数组中每条路径。
- 日志将多文件 `Read` 显示为并行读取卡片。
- 全量测试、类型检查和生产构建通过。
