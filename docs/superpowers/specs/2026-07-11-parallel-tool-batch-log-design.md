# 并行工具批次日志设计

## 目标

让执行日志准确表达同一模型响应中并行发出的多个工具调用，同时保留现有单项日志的文件预览、Diff、详情展开、状态和错误展示能力。

## 用户体验

- 单个工具调用继续按现有日志行显示。
- 同一模型响应包含两个或更多工具调用时，显示一张“并行执行 N 项”小卡片。
- 卡片执行中默认展开，完成后继续保持展开，由用户手动收起。
- 卡片内部按模型输出顺序显示原有日志行，不改变文件点击、Diff 点击、搜索详情或命令详情行为。
- 卡片标题显示整批总耗时：从该批最早调用开始，到最后一个调用结束。
- 外层执行日志继续沿用任务完成后自动收起的现有行为；用户重新展开外层日志时，批次卡片仍保持展开。

## 数据模型

AgentRunner 已将同一模型响应中的 `toolCallsArray` 使用 `Promise.all` 并行执行。为每个调用附加以下可选元数据：

- `batchId`：同一模型响应共享的稳定标识。
- `batchIndex`：调用在模型响应中的顺序。
- `batchSize`：该响应中的工具调用总数。

这些字段通过工具开始回调进入渲染进程，并保存在 `ToolCallState`。旧会话或单项调用没有批次字段时，继续走现有渲染路径。

## 渲染结构

统一时间线构建完成后，按 `batchId` 聚合同批工具项：

- `batchSize < 2`：直接渲染原日志行。
- `batchSize >= 2`：生成并行批次视图模型，包含批次状态、总耗时和原始子项。
- 不使用开始时间接近程度推断批次，避免快速连续的不同模型轮次被错误合并。

并行卡片只负责布局和批次摘要。子项继续复用 `LogItemRow`，避免复制点击和详情逻辑。

## 状态与耗时

- 任一子项为 `running` 时，批次状态为 `running`。
- 所有子项完成且至少一项失败时，批次状态为 `error`，标题补充失败数量。
- 所有子项成功时，批次状态为 `success`。
- 运行中耗时为当前时间减去最早 `startedAt`。
- 完成后耗时为最晚 `completedAt` 减去最早 `startedAt`。

## 展开状态

- 新批次首次出现时默认展开。
- 子项状态更新不得重置用户手动设置的批次展开状态。
- 批次完成时不自动收起。
- 外层日志的自动收起状态与批次展开状态相互独立。

## 兼容范围

- 主 Agent 与子 Agent 的工具批次均使用相同元数据语义；本次优先覆盖主执行日志，公共类型允许后续在子 Agent 卡片复用。
- 历史消息缺少批次字段时保持原有逐行展示。
- 不改变工具授权、执行顺序、并行策略或模型上下文记录。

## 验证

- 同一响应的三个读取调用显示为一张“并行执行 3 项”卡片。
- 同一响应的读取、搜索、浏览和命令显示为一张“并行执行 4 项”卡片。
- 单项调用不显示卡片。
- 文件、Diff 和详情点击行为与改动前一致。
- 总耗时采用整批起止时间，部分失败时显示失败状态。
- 执行中和完成后卡片保持展开，用户可手动收起。
- 旧消息没有批次字段时正常渲染。
