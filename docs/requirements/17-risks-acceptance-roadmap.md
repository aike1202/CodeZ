# 17. 风险、总验收与后续路线

> 模块：风险分析、第一版总验收标准、后续增强方向、结论

---

## 1. 风险分析

### 1.1 模型输出不稳定

风险：模型可能生成错误代码、错误命令或不完整 patch。

应对：

- 使用工具参数 schema；
- 使用 diff 审查；
- 增加验证命令；
- 限制自动循环次数；
- 失败时清晰反馈。

---

### 1.2 用户代码被误改

风险：Agent 修改不该修改的文件。

应对：

- 默认 diff 审查；
- 用户确认后写入；
- 应用前检查文件版本；
- 推荐用户使用 Git；
- 提供撤销或恢复提示。

---

### 1.3 API Key 泄露

风险：配置、日志或错误输出中泄露密钥。

应对：

- 安全存储；
- UI 脱敏；
- 日志脱敏；
- IPC 不随意传递明文密钥。

---

### 1.4 发送敏感代码到第三方模型

风险：私有代码、密钥、配置被发送到模型厂商。

应对：

- 显示隐私提示；
- 默认排除敏感文件；
- 支持本地模型；
- 支持用户确认发送范围；
- 支持关闭项目记忆。

---

### 1.5 命令执行风险

风险：Agent 执行破坏性命令。

应对：

- 命令执行前确认；
- 高风险命令二次确认；
- 命令超时；
- 限制工作目录；
- 禁止默认自动执行远程脚本。

---

## 2. 验收总标准

项目最终可认为完成第一版，当满足以下条件：

- [ ] 应用可以作为桌面端启动；
- [ ] 用户可以打开本地项目；
- [ ] 用户可以配置至少一个 OpenAI-compatible 模型；
- [ ] 用户可以与模型聊天；
- [ ] Agent 可以读取项目文件并回答项目问题；
- [ ] Agent 可以制定修改计划；
- [ ] Agent 可以生成代码变更 diff；
- [ ] 用户可以审查并应用 diff；
- [ ] Agent 可以在授权后运行测试或构建命令；
- [ ] 系统可以展示命令输出和验证结果；
- [ ] 会话和任务历史可以保存；
- [ ] API Key 不在 UI 和日志中明文泄露；
- [ ] 所有写文件和执行命令操作都有权限确认；
- [ ] 项目具备 `typecheck`、`test`、`build` 命令并可运行；
- [ ] 至少完成一个端到端演示：打开项目 → 提交任务 → 生成计划 → 修改代码 → 展示 diff → 应用修改 → 运行测试 → 输出总结。

---

## 3. 后续增强方向

第一版完成后，可以继续增强：

1. Tree-sitter 代码结构索引；
2. LSP 跳转定义和引用分析；
3. 向量数据库语义检索；
4. 多 Agent 协作：Planner、Coder、Reviewer、Tester；
5. 插件系统；
6. VS Code 插件版本；
7. 本地模型离线模式；
8. 自动生成 PR 描述；
9. 自动补测试；
10. 长任务后台执行；
11. 多工作区支持；
12. 更强的回滚机制；
13. 代码审查模式；
14. 需求文档/设计文档自动生成；
15. 团队协作和共享配置。

---

## 4. 结论

本项目的核心技术重点不在于训练大模型，而在于构建一个可靠的 Agent Coding 桌面系统。最关键的工程能力包括：

- 多模型 API 接入；
- Agent Loop；
- 工具调用系统；
- 项目上下文管理；
- 文件安全读写；
- diff 审查与应用；
- 命令执行和验证；
- 权限控制；
- 会话和项目记忆；
- 桌面端交互体验。

建议按照阶段 0 到阶段 9 逐步开发，保证每个阶段都有可运行版本、测试命令和验收标准。其中 Skills 与 MCP 放在最后阶段，作为核心 Agent Coding 流程稳定后的扩展能力。这样既适合作为学校项目，也适合作为长期演进为成熟 Agent Coding 产品的基础。
