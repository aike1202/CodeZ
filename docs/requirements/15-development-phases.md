# 15. 分阶段开发计划与验收

> 模块：阶段 0 到阶段 9，每个阶段都必须可运行、可测试、可验收

---

## 阶段原则

每个阶段都必须形成一个可运行版本，而不是等全部开发完成后再运行。每个阶段都需要有明确测试命令和验收标准。

---

## 阶段 0：项目初始化与基础工程

### 阶段目标

搭建 Electron + TypeScript + React 基础项目，使应用可以启动桌面窗口，并具备基础开发、构建、测试能力。

### 需求范围

- 初始化项目结构；
- 配置 Electron 主进程；
- 配置 React 渲染进程；
- 配置 TypeScript；
- 配置 Vite；
- 配置 ESLint/Prettier，可选；
- 配置 Vitest；
- 实现基础窗口；
- 显示欢迎页。

### 可运行测试

```bash
npm install
npm run dev
npm run typecheck
npm run test
npm run build
```

### 验收标准

- [ ] `npm run dev` 可以启动 Electron 桌面窗口；
- [ ] 窗口显示应用名称和欢迎页；
- [ ] `npm run typecheck` 通过；
- [ ] `npm run test` 至少运行一个示例测试并通过；
- [ ] `npm run build` 可以生成可打包产物；
- [ ] 主进程和渲染进程代码均使用 TypeScript。

---

## 阶段 1：项目打开、文件树与 Workspace 管理

### 阶段目标

用户可以选择本地项目目录，应用可以读取并展示文件树，建立 Workspace 概念。

### 需求范围

- 实现打开目录对话框；
- 实现 WorkspaceService；
- 扫描项目文件树；
- 过滤 `.git`、`node_modules` 等目录；
- 读取 `.gitignore`，可选；
- 显示最近打开项目；
- 支持点击文件查看内容；
- 识别基本项目类型。

### 可运行测试

```bash
npm run dev
npm run typecheck
npm run test
```

建议自动化测试：

- Workspace 路径合法性测试；
- 文件树过滤测试；
- 项目类型识别测试；
- 读取文件边界测试。

### 验收标准

- [ ] 用户可以从 UI 选择一个本地目录；
- [ ] 应用左侧展示该目录文件树；
- [ ] `node_modules`、`.git` 默认不展示；
- [ ] 点击文本文件可以查看内容；
- [ ] 点击二进制文件时提示不支持预览；
- [ ] 应用可以识别至少 Node.js 项目；
- [ ] 重启应用后可显示最近打开项目；
- [ ] 所有文件访问限制在 Workspace 内。

## 阶段 2：模型 Provider 配置与基础聊天

### 阶段目标

实现多模型 API 配置和基础聊天能力，用户可以配置 API Key，选择模型，与模型进行流式对话。

### 需求范围

- 实现 ModelProvider 抽象；
- 实现 OpenAI-compatible Provider；
- 支持配置 API Base URL、API Key、模型名；
- 支持测试连接；
- 支持模型流式回复；
- 实现聊天 UI；
- 会话消息本地保存；
- API Key 脱敏显示。

### 可运行测试

```bash
npm run dev
npm run typecheck
npm run test
```

手动联调：

```text
1. 打开设置页；
2. 配置一个 OpenAI-compatible API；
3. 点击测试连接；
4. 回到聊天页输入“你好”；
5. 确认模型流式返回。
```

### 验收标准

- [ ] 用户可以新增、编辑、删除模型 Provider；
- [ ] API Key 在 UI 中默认脱敏；
- [ ] 可以测试模型连接是否成功；
- [ ] 可以选择默认模型；
- [ ] 聊天页可以发送消息；
- [ ] 模型可以流式回复；
- [ ] 模型请求失败时显示清晰错误；
- [ ] 会话消息可以保存和恢复。

## 阶段 3：基础 Agent Loop 与只读工具调用

### 阶段目标

让 Agent 可以围绕用户任务读取项目上下文，调用只读工具分析项目，但不修改文件。

### 需求范围

- 实现 AgentRunner；
- 实现工具注册系统；
- 实现 `list_files`；
- 实现 `read_file`；
- 实现 `search_text`；
- 实现工具调用日志；
- 实现上下文构建；
- 支持 Agent 回答项目相关问题；
- 支持显示工具调用过程。

### 可运行测试

```bash
npm run dev
npm run typecheck
npm run test
```

建议测试用例：

- 工具参数 schema 校验；
- `read_file` 不能读取 Workspace 外文件；
- `search_text` 能返回匹配结果；
- 大文件读取会截断；
- 工具调用失败会返回错误。

### 验收标准

- [ ] 用户询问“这个项目是什么技术栈”时，Agent 能读取相关文件回答；
- [ ] 用户询问“有哪些 npm scripts”时，Agent 能读取 `package.json` 回答；
- [ ] 工具调用过程显示在聊天界面；
- [ ] 只读工具不需要每次弹窗确认，但必须记录日志；
- [ ] Agent 无法读取 Workspace 外文件；
- [ ] Agent 不会在本阶段修改任何文件。

## 阶段 4：任务计划与权限确认

### 阶段目标

对于复杂任务，Agent 能生成任务计划，并在执行涉及修改或命令的步骤前请求用户确认。

### 需求范围

- 实现任务识别；
- 实现 Planner；
- 实现任务计划 UI；
- 实现权限确认弹窗；
- 实现风险等级；
- 支持用户接受/拒绝计划；
- 支持用户中断任务；
- 支持任务状态保存。

### 可运行测试

```bash
npm run dev
npm run typecheck
npm run test
```

建议测试用例：

- 简单问答不进入任务计划；
- 修改类请求会生成计划；
- 用户拒绝计划后任务停止；
- 权限弹窗能正确返回允许/拒绝；
- 风险等级判断正确。

### 验收标准

- [ ] 用户输入“帮我添加一个功能”时，Agent 先生成计划；
- [ ] 计划中包含目标、步骤、涉及文件、验证方式；
- [ ] 用户可接受或拒绝计划；
- [ ] 用户拒绝后不会继续执行；
- [ ] 涉及写文件或命令执行的步骤必须请求确认；
- [ ] 任务状态可以在 UI 中查看。

## 阶段 5：代码修改、diff 预览与应用变更

### 阶段目标

Agent 可以生成代码修改建议，系统展示 diff，用户确认后应用到文件。

### 需求范围

- 实现变更对象模型；
- 实现 diff 生成；
- 实现 diff viewer；
- 实现 `write_file` 或 `edit_file`；
- 支持新增文件；
- 支持修改文件；
- 支持单文件接受/拒绝；
- 应用变更前检查文件是否变化；
- 应用后刷新文件树和文件内容。

### 可运行测试

```bash
npm run dev
npm run typecheck
npm run test
```

建议测试用例：

- 生成 diff 测试；
- 应用新增文件测试；
- 应用修改文件测试；
- 文件内容不匹配时拒绝应用；
- 用户拒绝 diff 后文件不变化。

### 验收标准

- [ ] Agent 可以对一个简单文件提出修改；
- [ ] 修改不会立即写入磁盘；
- [ ] UI 可以展示修改前后 diff；
- [ ] 用户接受后修改才写入文件；
- [ ] 用户拒绝后文件保持不变；
- [ ] 单个文件可以单独接受或拒绝；
- [ ] 应用变更失败时显示原因；
- [ ] 修改过程有日志记录。

## 阶段 6：命令执行、测试验证与修复循环

### 阶段目标

Agent 可以在用户授权后运行项目命令，读取输出，根据错误结果进行修复，并形成“修改-测试-修复”的闭环。

### 需求范围

- 实现 `run_command`；
- 实现命令确认弹窗；
- 实现命令输出面板；
- 支持命令超时和终止；
- 自动识别测试/构建命令；
- Agent 读取错误日志；
- 支持有限次数修复循环；
- 输出验证总结。

### 可运行测试

```bash
npm run dev
npm run typecheck
npm run test
```

在示例项目中手动验证：

```bash
npm test
npm run build
```

建议自动化测试：

- 允许命令执行测试；
- 拒绝命令执行测试；
- 命令超时测试；
- stdout/stderr 捕获测试；
- 退出码处理测试。

### 验收标准

- [ ] Agent 执行命令前显示命令、工作目录和目的；
- [ ] 用户确认后命令才运行；
- [ ] 命令输出实时显示；
- [ ] 命令完成后显示退出码；
- [ ] 命令超时可终止；
- [ ] 测试失败时 Agent 能读取错误并提出修复建议；
- [ ] 修复循环次数可配置；
- [ ] 最终总结包含运行了哪些命令、结果如何。

## 阶段 9：Skills 与 MCP 最终扩展

### 阶段目标

在核心 Agent Coding 流程稳定、桌面应用可安装运行之后，再接入 Skills 和 MCP。该阶段作为最终扩展阶段，目标是让应用具备可插拔能力包和外部工具协议接入能力。

### 放到最后的原因

Skills 和 MCP 依赖前序能力：

- 模型 Provider；
- Agent Loop；
- Tool Manager；
- Permission Manager；
- 命令执行；
- 本地存储；
- 日志审计；
- UI 设置页面；
- 安全策略。

如果过早实现，会增加核心链路复杂度。因此本项目先完成内置 Agent Coding 能力，最后再扩展 Skills 和 MCP。

### 需求范围：Skills 系统

- 实现 Skill 描述文件解析；
- 支持内置 Skills 和用户自定义 Skills；
- 支持 Skill 启用/禁用；
- 支持根据用户输入匹配 Skill；
- 支持 Skill Prompt 注入 Agent Runtime；
- 支持 Skill 声明所需工具权限；
- 支持 Skill 执行日志；
- 支持 Skill 管理 UI。

### 需求范围：MCP 系统

- 实现 MCP Server 配置管理；
- 支持启动、停止 MCP Server；
- 支持连接 MCP Server；
- 支持发现 MCP Tools；
- 支持发现 MCP Resources，可选；
- 将 MCP Tools 注册到内部 Tool Manager；
- MCP Tool 调用前走 Permission Manager；
- 记录 MCP 调用日志；
- 支持 MCP Server 错误处理和重连。

### 可运行测试

```bash
npm run typecheck
npm run test
npm run build
```

建议新增自动化测试：

- Skill 描述文件解析测试；
- Skill 启用/禁用测试；
- Skill 匹配测试；
- MCP 配置解析测试；
- MCP Tool 注册测试；
- MCP Tool 权限拦截测试；
- MCP Server 启动失败错误处理测试。

### 手动验收

```text
1. 启动应用；
2. 打开 Skills 管理页；
3. 启用一个内置 Skill；
4. 输入能触发该 Skill 的任务；
5. 确认 Agent 能按 Skill 工作流响应；
6. 打开 MCP 管理页；
7. 配置一个测试 MCP Server；
8. 启动并连接 MCP Server；
9. 确认 MCP Tools 出现在工具列表；
10. 触发一次 MCP Tool 调用；
11. 确认调用前有权限确认，调用后有日志。
```

### 验收标准

- [ ] 应用可以扫描并加载内置 Skills；
- [ ] 用户可以启用或禁用 Skill；
- [ ] Agent 可以根据用户输入匹配合适 Skill；
- [ ] Skill 可以向 Agent 注入工作流提示；
- [ ] Skill 所需工具权限可被识别和展示；
- [ ] 用户可以配置 MCP Server；
- [ ] 应用可以启动和停止 MCP Server；
- [ ] 应用可以发现 MCP Tools；
- [ ] MCP Tools 可以注册到 Tool Manager；
- [ ] MCP Tool 调用前必须经过权限确认；
- [ ] MCP 调用结果可以返回 Agent；
- [ ] MCP 调用失败时显示清晰错误；
- [ ] Skills 和 MCP 不影响基础 Agent Coding 流程。
