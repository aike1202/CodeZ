# 12. 非功能需求

> 模块：性能、稳定性、安全、隐私、兼容性

---

## 1. 性能需求

| 指标 | 要求 |
|---|---|
| 应用冷启动 | 目标小于 5 秒 |
| 打开中小型项目 | 目标小于 3 秒生成文件树 |
| 文件搜索 | 目标小于 2 秒返回前 100 条 |
| UI 响应 | 常规操作不阻塞主线程 |
| 模型流式响应 | 首 token 出现后立即渲染 |
| 大文件处理 | 超过阈值自动截断或分片 |

---

## 2. 稳定性需求

- 模型请求失败需要可重试；
- 工具调用失败需要返回清晰错误；
- 应用崩溃后应尽量恢复会话；
- 文件写入失败不能造成半写入损坏；
- 命令超时需要自动终止；
- 渲染进程异常不能影响本地文件安全；
- 长时间任务不应卡死 UI。

---

## 3. 安全需求

- Agent 默认只能访问用户打开的 Workspace；
- 读取 Workspace 外文件必须禁止或额外确认；
- 写入文件必须用户确认；
- 执行命令必须用户确认；
- API Key 需要安全存储；
- `.env`、密钥、证书文件默认不发送给模型；
- 网络请求需要明确来源；
- 高风险命令需要二次确认；
- 不允许 Agent 静默执行删除、推送、发布等操作。

---

## 4. 隐私需求

- 用户应知道哪些内容会发送到模型厂商；
- 提供“仅使用本地模型”的配置选项，可选；
- 支持清除会话历史；
- 支持清除项目记忆；
- 支持关闭遥测；
- 默认不上传整个项目；
- 敏感文件默认不进入上下文。

---

## 5. 兼容性需求

第一阶段优先支持：

- Windows 10/11；
- Node.js 18+ 或 20+；
- Git Bash / PowerShell / CMD 基础命令执行。

后续支持：

- macOS；
- Linux。

---

## 6. 可维护性需求

- 主进程、渲染进程、Agent Runtime、共享类型应分层清晰；
- IPC 接口必须集中管理；
- 模型 Provider 必须可扩展；
- 工具系统必须可注册新工具；
- 关键数据结构应有 TypeScript 类型；
- 核心逻辑应有单元测试。
