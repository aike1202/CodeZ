# 09. 代码修改与 Diff 审查

> 模块：变更生成、diff 预览、应用变更、冲突检测

---

## 1. 变更生成

### 功能编号

F23

Agent 修改代码时，系统应生成结构化变更。

每个变更包含：

- 文件路径；
- 变更类型：新增、修改、删除、重命名；
- 修改原因；
- diff 内容；
- 风险等级。

---

## 2. 变更对象要求

变更不应立即写入磁盘，而是先进入待审查状态。

推荐状态：

- pending；
- accepted；
- rejected；
- applied；
- failed。

---

## 3. Diff 预览

### 功能编号

F24

用户必须能在应用中查看 diff。

要求：

- 支持 side-by-side diff；
- 支持 inline diff；
- 支持按文件查看；
- 支持接受全部；
- 支持拒绝全部；
- 支持单文件接受；
- 支持复制 diff；
- 支持查看修改原因。

---

## 4. 应用变更

### 功能编号

F25

用户确认后系统才应用变更。

要求：

- 应用前检查文件是否被外部修改；
- 如果文件内容不匹配，提示冲突；
- 应用失败时回滚已应用部分，或至少给出失败清单；
- 应用后更新文件树和任务状态；
- 应用后记录日志。

---

## 5. 冲突检测

应用 diff 前需要校验：

- 文件是否仍存在；
- 文件 hash 是否与生成 diff 时一致；
- old content 是否匹配；
- 文件是否只读；
- 路径是否仍在 Workspace 内。

如果冲突，应停止应用并提示用户重新生成变更或手动处理。

---

## 6. 修改方式优先级

推荐优先级：

1. 精确编辑：old/new 字符串替换；
2. patch 应用；
3. 小文件完整重写；
4. 大文件完整重写仅在用户确认后允许。

---

## 7. 验收标准

- Agent 可以对一个简单文件提出修改；
- 修改不会立即写入磁盘；
- UI 可以展示修改前后 diff；
- 用户接受后修改才写入文件；
- 用户拒绝后文件保持不变；
- 单个文件可以单独接受或拒绝；
- 应用变更失败时显示原因；
- 修改过程有日志记录。
