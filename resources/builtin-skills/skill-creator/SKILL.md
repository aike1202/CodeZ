# Skill Creator（技能创建）

帮用户创建新技能，或改进已有技能。技能是一个带 YAML frontmatter 的 `SKILL.md`，可选地附带脚本、参考文档和资源文件。

高层流程：

- 弄清楚这个技能要做什么、大致怎么做
- 写出 `SKILL.md` 草稿（必要时创建 scripts/ references/ assets/ 子目录）
- 询问用户存到哪里（项目级还是全局），写入磁盘
- 给出一份简短的完成报告

你的任务是判断用户处在流程的哪一步，然后跳进去帮他推进。比如用户说"我想做一个做 X 的技能"，你就帮他把需求收窄、写草稿、组织好文件结构、存盘、给报告。

保持灵活：如果用户只想快速搞定，就别过度设计；如果用户想仔细打磨，就多迭代几轮。

## 与用户沟通

使用本技能的人技术背景差异很大。请根据上下文线索判断该用什么措辞，避免堆砌术语。拿不准某个词用户是否理解时，顺手用一句话解释一下。

---

## 一、明确意图

先理解用户想要什么。当前对话里可能已经包含了用户想沉淀的工作流（例如"把刚才这套流程做成技能"），先从对话历史里提取：用了哪些工具、步骤顺序、用户做过哪些纠正、输入输出格式。缺的部分再问用户，并在进入下一步前请他确认。

要问清楚：
1. 这个技能要让 AI 能做什么？
2. 什么时候应该触发它？（用户会用什么措辞/在什么场景）
3. 期望的输出格式是什么？
4. 是否需要附带脚本或参考资料？（见下面的"技能结构"）

主动追问边界情况、输入/输出格式、示例文件、成功标准、依赖。如果有可用的联网搜索能力，可以去查查同类技能、最佳实践，带着上下文回来，减少用户负担。

## 二、编写 SKILL.md

根据访谈填好这些部分：

- **name**：技能标识符（kebab-case，全小写；它同时是目录名和 `/触发名`）
- **description**：**这是技能能否被触发的首要机制**。要同时写清"做什么"和"什么时候用"。所有"何时使用"的信息都放这里，不要放正文。

  注意：模型有"欠触发"技能的倾向——该用时不用。为了对抗这一点，请把 description 写得稍微"主动"一点。例如与其写"构建仪表盘的方法"，不如写"构建仪表盘的方法。当用户提到仪表盘、数据可视化、内部指标，或想展示任何数据时，务必使用本技能，即使他没有明确说'仪表盘'。"

- **技能正文**：用祈使句写清指令。尽量解释每件事"为什么"重要——今天的模型很聪明，给它好的引导它能举一反三，而不是死板执行。如果你发现自己在写一堆全大写的 ALWAYS/NEVER 或极其死板的结构，那是个黄灯信号——尽量改成解释原因。先写草稿，再用新的眼光重看一遍、改进它。

## 三、技能结构

### 技能的构成

```
skill-name/
├── SKILL.md（必需）
│   ├── YAML frontmatter（name、description 必需）
│   └── Markdown 指令
└── 附带资源（可选）
    ├── scripts/    - 可执行代码，用于确定性/重复性任务
    ├── references/ - 按需加载进上下文的文档
    └── assets/     - 输出中会用到的文件（模板、图标、字体等）
```

如果技能需要脚本、模板或参考文档，就创建对应子目录。例如：
- 一个生成 docx 的技能，把 `create_docx.py` 放进 `scripts/`，在 SKILL.md 里让模型调用它——省得每次重新造轮子。
- 一个 HTML 邮件模板，放进 `assets/`。
- 一份 API 字段说明，放进 `references/`，在 SKILL.md 里指明何时去读它。

### 渐进式披露（Progressive Disclosure）

技能分三层加载：
1. **元数据**（name + description）——始终在上下文里（~100 词）
2. **SKILL.md 正文**——技能触发时进入上下文（理想 <500 行）
3. **附带资源**——按需加载（脚本可以直接执行而无需读入上下文）

关键做法：
- SKILL.md 尽量控制在 500 行内；接近上限时，加一层目录层级，并在 SKILL.md 里清楚指明"接下来该去读哪个文件"。
- 从 SKILL.md 清楚地引用其他文件，并说明何时该读。
- 大参考文件（>300 行）加个目录。

**按领域组织**：当一个技能支持多个领域/框架时，按变体拆分：
```
cloud-deploy/
├── SKILL.md（工作流 + 选择逻辑）
└── references/
    ├── aws.md
    ├── gcp.md
    └── azure.md
```
模型只读相关的那个参考文件。

### 写作风格

尽量用祈使句。解释"为什么"而非只说"做什么"。让技能通用，而不是死绑在具体例子上。

### 安全底线

技能里绝不能包含恶意代码、漏洞利用，或任何会危害系统安全的内容。技能的行为不应与它的描述相悖。不要配合创建具有误导性、或用于未授权访问/数据窃取等恶意目的的技能。（"扮演某个角色"这类是可以的。）

## 四、询问存放位置并写入

写入前**务必询问用户**技能存到哪里：
- **项目级**：`<项目根>/.skills/<skill-name>/`（仅当前项目可用，随项目走）
- **全局**：`~/.codez/skills/<skill-name>/`（所有项目可用）

如果当前没有打开项目，只能存全局。

然后创建 `<skill-name>/` 目录，写入 `SKILL.md` 及任何 scripts/ references/ assets/ 文件。

## 五、完成报告

技能写完存盘后，给用户一份**简短的完成报告**，包含：
1. **技能名与存放位置**（完整路径）
2. **已完成的工作**：用列表列出创建/修改了哪些文件（SKILL.md 及各子目录文件）
3. **如何使用**：在聊天里用 `/<skill-name>` 触发
4. **后续建议**（如有）：例如"可以补充测试用例""如需评测可用 scripts/ 下的工具"等

报告保持简明，别长篇大论。

## 可选：改进已有技能

如果用户是想改进一个已存在的技能，而非新建：
- **保留原名**：目录名和 frontmatter 的 `name` 都不要改。
- 读懂现有 SKILL.md，理解它当前的意图和不足。
- 针对用户反馈做修改：从反馈里**泛化**（技能要能用在很多场景，而不是只修好眼前这一个例子）；**保持精简**，去掉不发挥作用的内容；**解释为什么**。
- 改完后同样给一份简短的完成报告。

## 可选：评测与基准（进阶，通常不需要）

本技能目录下的 `scripts/`、`eval-viewer/`、`agents/`、`references/` 里带有一套完整的量化评测工具（跑测试用例、对比基准、生成审查视图、优化 description 触发率）。这套流程较重，依赖 Python 环境，且是为大规模严肃评测设计的。

**默认不要主动跑这套评测流程。** 只有当用户明确要求"评测这个技能""跑基准""优化触发率"时，才去读对应脚本并按其说明执行。日常创建一个技能不需要它。
