## 需求概述

**一句话描述**：实现多文件修改的一键备份与回滚事务服务，并对流式对话的消息上下文窗口实施智能裁剪与压缩，以防 Token 爆掉和代码库半损坏。

**业务背景**：
1. **修改易出错且难回滚**：当前 Agent 在执行代码修改时，直接对物理文件进行写入。当执行复杂的跨多文件协同修改时，中途某一步（例如编译失败、语法错误）如果发生错误，代码库将处于破损状态，需要开发者手动回滚，极大打折扣了 Agent 的可用性和容错性。
2. **上下文管理粗暴且容易溢出**：当前 `App.tsx` 使用极其简单的 `chatMessages = [...currentMsgs.slice(-20)]` 来剪切消息记录。这会导致：① 系统预设 Prompt 被裁剪丢弃；② 如果之前的工具调用输出了海量文本（如 `get_project_snapshot` 返回了几万字），这 20 条消息也会瞬间超出 LLM 的 Token 上限并导致 API 请求被拒，必须进行智能裁剪。

**预期价值**：
1. **高安全性**：Agent 修改代码将以“事务”形式受控执行，当修改后编译报错或 Agent 自主判定失败时，支持一键撤销（Rollback）回滚所有受影响文件。
2. **高稳定性**：支持智能截断长 Tool 输出，保障大上下文请求不会遭遇 Token 溢出错误，提高大模型 API 请求的稳定性和会话持久度。

## 功能需求

### 核心功能（必须实现）

#### F1: 跨文件修改事务管理服务（Edit Transaction Service）
* **F1.1: 自动备份机制 (Backup)**:
  * 当 Agent 触发修改物理文件（`write_to_file` 或 `replace_file_content`）前，系统检测该文件是否在当前“修改事务”中已有备份。
  * 如果是本次事务第一次修改此文件，自动将其原内容拷贝并备份在应用的数据目录（`userData` 下的 `backup/<session_id>/` 临时文件夹内）。
* **F1.2: 事务回滚工具 (Rollback Tool)**:
  * 注册一个新的 Builtin Tool `rollback_last_edit`。
  * 参数：无，或者接受 `reason`（可选，说明回滚的原因）。
  * 效果：将最近一轮修改事务备份的所有文件原封不动复制回去，覆盖当前的破损物理文件，确保代码库完全回退到修改前。
* **F1.3: 事务提交与清理 (Commit & Clean)**:
  * 当一轮 Agent 生成循环正常结束并展示给用户，或者进入下一个会话交互周期时，自动提交本轮事务，删除临时文件夹中的备份。

#### F2: 智能消息上下文窗口管理器（Smart Context Manager）
* **F2.1: 系统提示词永久锚定 (Anchor System Prompt)**:
  * 保证对话历史中最前面的 System Prompt 永远保留，任何裁剪算法均不得裁剪掉首条 System 消息。
* **F2.2: 工具执行结果的智能截断 (Truncate Tool Response)**:
  * 构建发往大模型的 Payload 时，对历史 Tool 类型的消息内容进行字数控制。
  * 若某条 Tool 执行结果字符数超过阈值（如超过 3000 字符，例如超大文件读取或长目录树），系统将其中间内容裁剪，并替换为 `[Output Truncated... Actual size: XXX chars]` 的尾部摘要。
  * 保证大模型依然能获得 Tool 的头部与尾部关键逻辑，同时缩减 Token 开销。

## 非功能需求

### 性能要求
* 备份与回滚的 IO 延迟限制在 150ms 以内（纯本地小文件复制）。
* 上下文裁剪计算在 5ms 内完成。

### 兼容性要求
* 全面支持多进程 Electron 环境，Main 进程管理文件事务，Preload API 导出回滚功能。
