# 需求分析：搜索代码逻辑优化

## 业务背景
在当前项目的智能体（Agent）工具箱中，`SearchTextTool` 使用了简单的深度优先 JS 遍历来搜索文件内容。然而，其中硬编码了一个扫描文件数量的上限（`const limit = 200`）。当项目体积稍大（文件超过 200 个）时，工具在搜完 200 个文件后就会强制退出，导致经常出现“搜索不到存在的内容”（如搜索 `addNote` 毫无结果）的 Bug。

为了回答您的问题：“我是如何快速定位代码的”——我使用的是底层环境提供的 `grep_search` 工具，它封装了极速的文本搜索工具（如 `ripgrep`），并且完全无文件数量上限，天然忽略无关的二进制和第三方库文件。

为了让您的 Agent 也能具备这样优秀的搜索能力，我们需要优化 `SearchTextTool`。

## 功能需求
- [ ] **F1**: 移除原有的 200 个文件扫描上限。
- [ ] **F2**: 引入高效的系统级搜索方案。首选利用 `git grep` 命令（因为它速度极快，且天然遵循 `.gitignore` 的排除规则），这样就不需要手动排查 `node_modules` 等目录。
- [ ] **F3**: 当 `git grep` 失败（例如当前不在 git 仓库或未安装 git）时，降级回退到优化的 Node.js 遍历逻辑。
  - 降级方案中需包含常见文件后缀过滤或扩展名白名单（如只读 `.ts`, `.js`, `.json`, `.md` 等文本文件）。
  - 处理搜索关键字中的特殊字符（正则转义）。
- [ ] **F4**: 限制输出结果的长度（例如最多返回 50-100 条结果，或截断过长字符），以防超大的搜索结果直接撑爆大模型的上下文窗口。

## 验收标准
- [ ] **AC1**: 当在根目录下搜索某个处于较深目录下的代码词汇（如 `addNote`）时，能够成功返回匹配行。
- [ ] **AC2**: 搜索响应时间控制在毫秒或1秒级别（借助 `git grep`）。
- [ ] **AC3**: 工具能正确防范超大结果把终端卡死。
