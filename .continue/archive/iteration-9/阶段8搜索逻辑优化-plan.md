## 整体技术与架构总览
当前 `SearchTextTool` 存在扫描文件数量上限，导致搜索不到内容的 Bug，同时单纯的 Node.js 遍历在庞大代码库中性能较差。
本阶段的技术主轴是：**分级搜索策略（系统级高并发搜索 + 语言级遍历回退）**。
- **阶段一（优先策略）**：引入 `git grep -inI -E` 搜索。它速度极快并天然遵循 `.gitignore` 的排除规则，非常适合绝大多数使用 Git 进行版本控制的项目。
- **阶段二（降级策略）**：对于非 Git 仓库，或者 `git grep` 执行失败的场景，平滑回退到优化的 Node.js 遍历逻辑，通过移除文件数硬限制、引入后缀名白名单及安全的大小限制（防止大文件卡死），并并发读取文件，提高搜索效率。
- **结果保护**：统一在最终输出时对匹配条数做截断（上限比如 50~100 条），防止撑爆大模型的上下文。

## 阶段与任务大纲

**目标**：重构 `SearchTextTool` 的实现，利用 `git grep` 加速搜索，并提供完善的降级方案，彻底解决“搜索不到内容”的 Bug，并保护模型上下文。

✅ 第一阶段 · 优化 SearchTextTool 核心逻辑
  ✅ 1、[任务 T1] 移除硬编码文件扫描数量上限
     - 极简说明：在 `src/main/tools/builtin/SearchTextTool.ts` 中，移除类似 `limit = 200` 的逻辑，改为控制总匹配条数（如 `maxResults = 50`）。
  ✅ 2、[任务 T2] 引入 `git grep` 优先搜索
     - 极简说明：通过 `child_process.exec` 执行 `git grep -inI -E "<query>"`, 匹配则直接返回结果。
  ✅ 3、[任务 T3] 实现优化的 Node.js 降级搜索
     - 极简说明：当 `git grep` 失败时进行降级，限制后缀白名单及单个文件大小，优化遍历读取速度。
  ✅ 4、[任务 T4] 统一结果输出截断与兜底
     - 极简说明：确保搜索结果截断以防止输出过大，并在最后提示 `... (more results truncated)`。

### 验收&测试点
  ⏳ 1、[非 Git 仓库搜索测试]：在没有初始化 git 的项目目录执行搜索，验证 Node.js 降级方案能否正常找到深层代码，并且搜索时间合理。
  ⏳ 2、[搜索条数截断测试]：搜索常见词汇（如 `const` 或 `import`），验证最终返回结果是否受限在设定的条数。
