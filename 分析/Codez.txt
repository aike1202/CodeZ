Open with the most characteristic thing in the subject's world, in whatever form makes sense for it: a headline, an image, an animation, a live demo, an interactive moment. Be deliberate with your choice: a big number with a small label, supporting stats, and a gradient accent is the template answer, only use if that's truly the best option.\n\nTypography carries the personality of the page. Pair the display and body faces deliberately, not the same families you would reach for on any other project, and set a clear type scale with intentional weights, widths, and spacing. Structural devices, numbering, eyebrows, dividers, labels, should encode something true about the content, not decorate it. Many generic designs use numbered markers (01 / 02 / 03), but that's only appropriate if the content actually is a sequence - like a real process or a typed timeline where order carries information the reader needs. Think about where and if animation can serve the subject: a page-load sequence, a scroll-triggered reveal, hover micro-interactions, ambient atmosphere. Maximalist directions need elaborate execution; minimal directions need precision in spacing, type, and detail. Elegance is executing the chosen vision well.\n\nConsider written content carefully. It's faster and works anywhere.\r\n\r\n## When to Use\r\n\r\n- User has a frontend codebase and wants to extract or document its design system\r\n- User wants to migrate a project's visual identity into Stitch\r\n- User asks to \"audit the styling\" or \"understand the design language\" of a repo\r\n- User wants to create a DESIGN.md from existing source code\r\n- The app can't be built/rendered but the source is available\r\n- User wants to unify or reconcile inconsistent styles across a codebase\r\n\r\n## Prerequisites\r\n\r\n- Access to the frontend project's source directory\r\n- No build or runtime dependencies needed — this skill reads source files only\r\n\r\n---\r\n\r\n## Workflow\r\n\r\n### Phase 1: Project Discovery\r\n\r\nStart by understanding what you're working with. This determines which\r\nextraction patterns to use.\r\n\r\n#### 1. Detect the Framework and Stack\r\n\r\nScan the project root for telltale files:\r\n\r\n| Signal File | Framework / Tool |\r\n|:---|:---|\r\n| `package.json` with `react` | React / Next.js |\r\n| `package.json` with `vue` | Vue / Nuxt |\r\n| `package.json` with `svelte` | Svelte / SvelteKit |\r\n| `package.json` with `@angular/core` | Angular |\r\n| `tailwind.config.js/ts` | Tailwind CSS |\r\n| `postcss.config.js` | PostCSS pipeline |\r\n| `styled-components` or `@emotion` in deps | CSS-in-JS |\r\n| `.css` / `.scss` / `.less` files only | Plain CSS / SASS |\r\n| `theme.js` / `theme.ts` / `tokens.js` | Design token files |\r\n\r\nRead `package.json` first — it reveals the framework, CSS tooling, and any\r\ndesign-token libraries (e.g., `style-dictionary`, `@chakra-ui/react`,\r\n`@mui/material`, `ant-design`). This context tells you *where* to look for\r\nstyling information.\r\n\r\n#### 2. Map the Source Tree\r\n\r\nIdentify the key directories and files you'll analyze:\r\n\r\n```\r\nsrc/\r\n├── components/     ← Component-level styles\r\n├── styles/         ← Global stylesheets\r\n├── theme/          ← Theme definitions, tokens\r\n├── assets/         ← Fonts, images\r\n├── app.css         ← Root styles\r\n└── index.css       ← Entry CSS\r\n```\r\n\r\nAlso check for:\r\n- `tailwind.config.js` / `tailwind.config.ts` — Custom colors, fonts, spacing\r\n- `globals.css` / `global.css` — CSS custom properties (variables)\r\n- Any `theme.*` or `tokens.*` files\r\n- Component library config (e.g., `chakra-theme.ts`, `vuetify.config.ts`)\r\n\r\n#### 3. Read Framework-Specific Guidance\r\n\r\nConsult the appropriate reference for extraction patterns:\r\n\r\n- **React / Next.js / Tailwind** → [references/react-tailwind.md](references/react-tailwind.md)\r\n- **Vue / Nuxt** → [references/vue.md](references/vue.md)\r\n- **Svelte / SvelteKit** → [references/svelte.md](references/svelte.md)\r\n- **Angular** → [references/angular.md](references/angular.md)\r\n- **Plain CSS / SASS / Less** → [references/plain-css.md](references/plain-css.md)\r\n\r\nThese references contain framework-specific patterns for locating colors,\r\ntypography, spacing, and component styles. Read the one that matches before\r\nproceeding.\r\n\r\n---\r\n\r\n### Phase 2: Deep Extraction\r\n\r\nWork through each design dimension systematically. For each one, gather raw\r\ndata from the source files, then synthesize it into descriptive language.\r\n\r\nThe goal isn't to dump every CSS property — it's to understand the *intent*\r\nbehind the styling choices and describe them in human, editorial language that\r\nanother designer (or Stitch) can use to recreate the same visual feel.\r\n\r\n#### 1. Visual Theme & Atmosphere\r\n\r\nRead the broadest styling first to understand the overall mood:\r\n\r\n- **Root background**: What's the `body` or root element background? Light\r\n  cream (#f-range) signals airy/clean; dark (#0-#2 range) signals moody/dramatic.\r\n- **Whitespace philosophy**: Are spacing values generous (32px+) or tight?\r\n  Check padding/margin values on root containers, section wrappers, and card components.\r\n- **Density**: Count the components per page/section. Few with space = minimal;\r\n  many packed tight = information-dense.\r\n- **Color temperature**: Are the neutrals warm (creams, tans) or cool (blue-grays, slates)?\r\n- **Overall feel**: Synthesize into 1-2 rich sentences that capture the mood.\r\n\r\nLook for these signals in the source:\r\n\r\n| Source Location | What It Tells You |\r\n|:---|:---|\r\n| Root `background-color` or Tailwind `bg-*` on layouts | Overall lightness/darkness |\r\n| Spacing scale in Tailwind config or CSS vars | Whitespace philosophy |\r\n| Number of components vs. wrapper padding | Density |\r\n| Custom property naming (`--warm-*` vs `--cool-*`) | Color temperature intent |\r\n| Comments in theme files | Design intent in the developer's own words |\r\n\r\n#### 2. Color Palette & Roles\r\n\r\nExtract every unique color from the codebase and assign functional roles.\r\nSearch across all layers:\r\n\r\n**Where to find colors:**\r\n\r\n| Layer | What to Search |\r\n|:---|:---|\r\n| CSS custom properties | `--color-*`, `--primary`, `--bg-*` |\r\n| Tailwind config | `theme.extend.colors` |\r\n| Theme/token files | Color objects, palettes |\r\n| Component styles | `background-color`, `color`, `border-color` |\r\n| Inline/scoped styles | `bg-*`, `text-*` classes in templates |\r\n| CSS-in-JS theme objects | `colors`, `palette` keys |\r\n\r\n**How to organize:** Group colors by function, not by hue:\r\n\r\n1. **Primary Foundation** — Background and surface colors\r\n2. **Accent & Interactive** — CTA buttons, active states, links\r\n3. **Typography & Text Hierarchy** — Primary, secondary, tertiary text\r\n4. **Functional States** — Success, error, warning, info\r\n\r\nFor each color, create a descriptive name that evokes the color's character\r\nrather than its raw hex value:\r\n\r\n- ❌ `#294056` → \"Blue\"\r\n- ✅ `#294056` → **\"Deep Muted Teal-Navy\"** — Primary CTA, active navigation\r\n\r\n**Deduplication matters.** Codebases often have near-duplicate colors (e.g.,\r\n`#333` and `#2C2C2C`). Consolidate them under one name that best represents\r\nthe intended color.\r\n\r\n#### 3. Typography Rules\r\n\r\nExtract the complete typographic system:\r\n\r\n**Font families:**\r\n- Check CSS `font-family`, Tailwind `fontFamily`, Google Fonts links, or\r\n  local `@font-face` declarations.\r\n- Note the **character** of each font: geometric vs humanist, serif vs sans,\r\n  the feeling it evokes.\r\n\r\n**Type scale (hierarchy):**\r\n- Find every heading level (H1-H6) and body text, noting:\r\n  - `font-size` (in rem or px)\r\n  - `font-weight` (numeric value + descriptive name)\r\n  - `letter-spacing` (and why — elegance? compactness?)\r\n  - `line-height` (generous for readability? tight for display?)\r\n- Map component usage: Which heading level do product cards use? What about\r\n  hero sections?\r\n\r\n**Spacing principles:**\r\n- How does text spacing relate to the overall spacing scale?\r\n- Letter-spacing patterns on headings vs body\r\n- Line-height philosophy (generous/relaxed for body, tighter for display)\r\n\r\n#### 4. Component Stylings\r\n\r\nAnalyze the 4-5 most important UI primitives:\r\n\r\n**Buttons:**\r\n- Corner radius (and what it communicates — playful? professional? minimal?)\r\n- Color scheme for primary, secondary, and ghost variants\r\n- Hover/focus/active states and transition timing\r\n- Padding ratios (horizontal vs vertical)\r\n\r\n**Cards / Containers:**\r\n- Corner radius (often different from buttons — slightly rounder)\r\n- Shadow strategy: flat, subtle hover shadows, or always elevated?\r\n- Border treatment: hairline borders, colored accents, or none?\r\n- Internal padding (generous or compact?)\r\n- Image treatment within cards (full-bleed, padded, rounded?)\r\n\r\n**Navigation:**\r\n- Layout pattern (horizontal bar, vertical sidebar, drawer)\r\n- Typography treatment (uppercase, letter-spacing, weight)\r\n- Active/hover state indicators (underline, color, background)\r\n- Mobile behavior (hamburger, bottom nav, drawer)\r\n\r\n**Inputs & Forms:**\r\n- Border style and focus state behavior\r\n- Corner style consistency with buttons\r\n- Padding and touch-target sizing\r\n\r\n**Domain-Specific Components:**\r\n- Identify 1-2 components unique to this project (e.g., product cards,\r\n  dashboard widgets, chat bubbles) and describe their styling patterns.\r\n\r\n#### 5. Layout Principles\r\n\r\nExtract the structural system:\r\n\r\n**Grid & Structure:**\r\n- Max content width (from `max-width` on containers)\r\n- Column system (CSS Grid, Flexbox patterns, defined breakpoints)\r\n- Responsive breakpoints (from media queries or Tailwind config)\r\n\r\n**Whitespace Strategy:**\r\n- Base spacing unit (8px grid? 4px? custom?)\r\n- Section margins (how much space between major sections)\r\n- Edge padding (page margins at different breakpoints)\r\n\r\n**Alignment & Visual Balance:**\r\n- Text alignment patterns (centered heroes, left-aligned body)\r\n- Image-to-text ratios\r\n- Visual weight distribution\r\n\r\n**Responsive Behavior:**\r\n- Mobile-first or desktop-first?\r\n- How do grids collapse? Padding scale?\r\n- Touch target sizing\r\n\r\n#### 6. Stitch Generation Notes\r\n\r\nSynthesize the extraction into actionable prompts for Stitch:\r\n\r\n- **Atmosphere language**: Translate the mood into natural descriptors\r\n- **Color references**: List colors by descriptive name + hex\r\n- **Component prompts**: Write 2-3 example prompts that would recreate\r\n  key components in Stitch\r\n- **Iteration guidance**: Tips for refining screens in this design system\r\n\r\n---\r\n\r\n### Phase 3: Write the DESIGN.md\r\n\r\nAssemble everything into the standard DESIGN.md format. Place it at\r\n`.stitch/DESIGN.md` in the project directory (create the `.stitch/` directory\r\nif it doesn't exist).\r\n\r\n> [!IMPORTANT]\r\n> You **MUST** include the YAML frontmatter at the top of the file with `name` and `colors` mapping, exactly as shown in the example at [examples/DESIGN.md](examples/DESIGN.md). This structured data is required for other skills to parse the design system.\r\n>\r\n> Failure to include this YAML block with at least the core color tokens is a failure to use this skill correctly.\r\n\r\nUse the format from the example at [examples/DESIGN.md](examples/DESIGN.md) as your template. The file must start with the YAML block, followed by the markdown sections:\r\n\r\n```markdown\r\n# Design System: [Project Name]\r\n**Project ID:** [If known, otherwise omit]\r\n\r\n## 1. Visual Theme & Atmosphere\r\n[Rich 2-paragraph description of mood, philosophy, and key characteristics]\r\n\r\n## 2. Color Palette & Roles\r\n### Primary Foundation\r\n### Accent & Interactive\r\n### Typography & Text Hierarchy\r\n### Functional States\r\n\r\n## 3. Typography Rules\r\n### Hierarchy & Weights\r\n### Spacing Principles\r\n\r\n## 4. Component Stylings\r\n### Buttons\r\n### Cards & [Domain-Specific Containers]\r\n### Navigation\r\n### Inputs & Forms\r\n### [Domain-Specific Components]\r\n\r\n## 5. Layout Principles\r\n### Grid & Structure\r\n### Whitespace Strategy\r\n### Alignment & Visual Balance\r\n### Responsive Behavior & Touch\r\n\r\n## 6. Design System Notes for Stitch Generation\r\n### Language to Use\r\n### Color References\r\n### Component Prompts\r\n### Incremental Iteration\r\n```\r\n\r\n---\r\n\r\n### Phase 4: Integration (Optional)\r\n\r\nIf the user wants to push the design system into Stitch:\r\n\r\n1. Hand off to the `manage-design-system` skill for the MCP create/update calls\r\n2. The DESIGN.md you wrote is the input — the manage-design-system skill handles\r\n   the Stitch API integration\r\n\r\nIf the user just wants the document, you're done after Phase 3.\r\n\r\n---\r\n\r\n## Quality Checklist\r\n\r\nBefore delivering the DESIGN.md, verify:\r\n\r\n- [ ] Every color has a descriptive name, hex code, and functional role\r\n- [ ] Typography includes font family, character description, and full hierarchy\r\n- [ ] Component styles describe shape, color, states, and transitions\r\n- [ ] Layout includes max-width, grid, breakpoints, and spacing strategy\r\n- [ ] Stitch generation notes use natural language, not CSS syntax\r\n- [ ] The atmosphere section reads like editorial copy, not technical docs\r\n- [ ] Near-duplicate colors are consolidated\r\n- [ ] The document captures the *intent* behind styling, not just raw values\r\n\r\n## Tips for Better Extraction\r\n\r\n- **Read comments and commit messages.** Developers often document design\r\n  intent in code comments (`/* hero section — breathable */`) and commit\r\n  messages. These are gold for understanding the *why*.\r\n- **Check for design-token libraries.** If the project uses `style-dictionary`,\r\n  `@tokens-studio`, or similar, these files are the most authoritative\r\n  source of design values.\r\n- **Theme files are higher-signal than component styles.** A `theme.ts` that\r\n  defines a palette tells you the intended design system; scattered inline\r\n  styles in components tell you what actually shipped. Both matter, but\r\n  start from the theme.\r\n- **Tailwind config is a design system.** If a project has a customized\r\n  `tailwind.config.js`, that *is* the design system — extract from it first,\r\n  then spot-check components for overrides.\r\n- **CSS custom properties are intentional.** If a developer defined\r\n  `--brand-primary`, they're telling you this is a design token. Present the options clearly, **recommend Strategy A** as the preferred default, and **provide a brief pros/cons summary** for each option to help them make an informed decision.\r\n\r\n| | Strategy A (Puppeteer) | Strategy B (Browser Subagent) |\r\n| :--- | :--- | :--- |\r\n| **When** | App runs locally, no auth wall | Need to interact with page first (click, fill forms) |\r\n| **Fidelity** | **Highest — computed styles resolved** | High — rendered DOM |\r\n| **Setup** | **Zero — no mock needed** | Zero — no mock needed |\r\n| **Framework** | **Any** | Any |\r\n| **Output** | **Writes to file — no size limit** | May truncate in agent context |\r\n\r\n> [!WARNING]\r\n> **Checkpoint — User Confirmation Required.**\r\n> You **MUST** ask the user which strategy they prefer before proceeding.\r\n> Present the comparison table above, recommend Strategy A as the default, and\r\n> wait for explicit approval. Do **NOT** make the decision yourself or proceed\r\n> until the user confirms.\r\n\r\n***\r\n\r\n## Strategy A: Puppeteer Snapshot (Recommended)\r\n\r\nLaunches headless Chrome, captures the fully rendered DOM, and produces a self-contained HTML file with all CSS inlined and images as base64. Works with **any framework** — no MockPage.jsx needed.\r\n\r\n### Prerequisites\r\n\r\n- App running locally (e.g., `npm run dev`)\r\n- Node.js with `puppeteer` available (check: `node -e \"require('puppeteer')\"`)\r\n\r\n### Workflow\r\n\r\n1.  **Start the App** and note the port.\r\n\r\n    > [!WARNING]\r\n    > **Checkpoint — User Confirmation Required.**\r\n    > After starting the local server, you **MUST** pause and ask the user for\r\n    > confirmation before running the snapshot script or launching a browser\r\n    > subagent. Report the URL and port to the user so they can verify the app\r\n    > is running and rendering correctly. Do **NOT** proceed to the snapshot\r\n    > step until the user confirms.\r\n\r\n2.  **Run the Snapshot Script**:\r\n    ```bash\r\n    npx tsx <SKILL_DIR>/scripts/snapshot.ts \\\r\n      --url http://localhost:5173 \\\r\n      --output .stitch/home.html \\\r\n      --wait 2000\r\n    ```\r\n\r\n3.  **Multiple pages** — run once per route:\r\n    ```bash\r\n    npx tsx <SKILL_DIR>/scripts/snapshot.ts \\\r\n      --url http://localhost:5173 --output .stitch/home.html --wait 2000\r\n    npx tsx <SKILL_DIR>/scripts/snapshot.ts \\\r\n      --url http://localhost:5173/pricing --output .stitch/pricing.html --wait 2000\r\n    npx tsx <SKILL_DIR>/scripts/snapshot.ts \\\r\n      --url http://localhost:5173/dashboard --output .stitch/dashboard.html --wait 2000 --html-class dark\r\n    ```\r\n\r\n### Script Flags\r\n\r\n| Flag | Default | Description |\r\n| :--- | :--- | :--- |\r\n| `--url` | *(required)* | URL to capture |\r\n| `--output` | *(required)* | Output file path |\r\n| `--wait` | `1000` | Extra wait (ms) after network idle. Increase for lazy-loading apps. |\r\n| `--viewport` | `1280x800` | Viewport size as `WIDTHxHEIGHT` |\r\n| `--html-class` | — | Class(es) for `<html>` element (e.g., `dark`) |\r\n| `--remove-fixed` | `false` | Remove fixed/sticky elements (cookie banners, chat widgets) |\r\n| `--full-height` | `false` | Resize viewport to full scroll height |\r\n| `--title` | — | Override page title |\r\n\r\n### What It Does Automatically\r\n\r\n- Inlines all `<link rel=\"stylesheet\">` → `<style>` blocks\r\n- Converts `<img>` `src` **and `srcset`** → base64 data URIs (skips fonts)\r\n- Inlines `<source srcset>` URLs as base64\r\n- Removes failed/dead `srcset` entries so the browser falls back to the inlined `src`\r\n- Removes `<script>` tags, Vite overlay, Next.js dev indicators\r\n- Resolves relative CSS `url()` paths before inlining\r\n\r\n### Framework Notes\r\n\r\n| Framework | Notes |\r\n| :--- | :--- |\r\n| **React + Vite** | Works out of the box. `--wait 1000`. |\r\n| **Next.js** | `--wait 3000` for SSR hydration. URL: `http://localhost:3000`. `<img srcset>` from `/_next/image` is auto-inlined as base64. |\r\n| **Vue / Nuxt** | Works out of the box. |\r\n| **Svelte / SvelteKit** | Works out of the box. |\r\n| **Storybook** | Use story URL: `--url http://localhost:6006/?path=/story/...` |\r\n| **SSR (Webpack)** | May need longer `--wait`. |\r\n\r\n### Troubleshooting\r\n\r\n| Issue | Solution |\r\n| :--- | :--- |\r\n| Images missing | Increase `--wait` |\r\n| Images show as broken after server stops | Verify `srcset` was inlined — check log for \"Inlined N images\". The browser subagent gives you full control but output may truncate for large pages.\r\n\r\n### Workflow\r\n\r\n1.  **Start the App** locally.\r\n2.  **Navigate** using a browser subagent.\r\n3.  **Interact** as needed (click, scroll, fill forms).\r\n4.  **Extract DOM**: `document.documentElement.outerHTML`\r\n\r\n    > [!WARNING]\r\n    > Large pages may truncate. It requires manually flattening React components into a single JSX file. **Prefer Strategy A whenever possible.**\r\n\r\n### When to Use\r\n\r\n- App can't run locally at all\r\n- Page requires auth <SKILL_DIR>/scripts/extract_inline_html.ts \\\r\n  --index-css src/css/App.css \\\r\n  --extra-css index.html \\\r\n  --outdir .stitch \\\r\n  --page src/MockPage.jsx:Page.html:\"Page Present AI Insights\r\n\r\nAfter any tool call, always surface the `outputComponents` (Text Description and\r\nSuggestions) to the user.\r\n\r\nSee [examples/enhanced-prompt.md](examples/enhanced-prompt.md) for a full\r\nbefore/after prompt enhancement example.\r\n\r\n--------------------------------------------------------------------------------\r\n\r\n## Steps\r\n\r\n### Determine the Mode\r\n\r\nDecide which flow to use based on the user's request:\r\n\r\n- User wants to create from a text description → **Generate from Text** flow\r\n- User provides an image, screenshot, or mockup → **Generate from Image** flow\r\n- User wants to modify an existing screen → **Edit** flow\r\n- User wants layout/color/content variations → **Generate Variants** flow\r\n\r\n---\r\n\r\n### Generate from Text Flow (New Screen)\r\n\r\n#### 1. Enhance the User Prompt\r\n\r\nApply the Prompt Enhancement Pipeline above.\r\n\r\n#### 2. Identify the Project\r\n\r\nUse `list_projects` to find the correct `projectId` if it is not already known.\r\n\r\n#### 3. Generate the Screen\r\n\r\nCall the `generate_screen_from_text` tool with the enhanced prompt and the\r\n`designSystem` ID (if found in Step 1).\r\n\r\n```json\r\n{\r\n  \"projectId\": \"...\",\r\n  \"prompt\": \"[Your Enhanced Prompt]\",\r\n  \"designSystem\": \"assets/...\", // Optional: Pass if found in Step 1\r\n  \"deviceType\": \"DESKTOP\"  // Options: MOBILE, DESKTOP, TABLET\r\n}\r\n```\r\n\r\n#### 4. Present AI Feedback\r\n\r\nAlways show the text description and suggestions from `outputComponents` to the\r\nuser.\r\n\r\n#### 5. Download Design Assets\r\n\r\nAfter generation, download the HTML and screenshot urls from `outputComponents`\r\nto the `.stitch/designs` directory.\r\n\r\n- **Naming**: Use the screen ID or a descriptive slug for the filename.\r\n- **Tools**: Use `curl -o` via `run_command` or similar.\r\n- **Directory**: Ensure `.stitch/designs` exists.\r\n\r\n#### 6. Review and Refine\r\n\r\n- If the result is not exactly as expected, continue with the **Edit** flow\r\n  to make targeted adjustments.\r\n- Do NOT re-generate from scratch unless the fundamental layout is wrong.\r\n\r\n---\r\n\r\n### Generate from Image Flow (Image/Mockup → Design)\r\n\r\nUse this flow when the user provides an image, screenshot, or design mockup to\r\nrecreate in Stitch.\r\n\r\n#### 1. Identify the Project\r\n\r\nUse `list_projects` to find the correct `projectId`. If no suitable project\r\nexists, create one using `create_project`.\r\n\r\n#### 2. Upload the Image\r\n\r\nDelegate to the **upload-to-stitch** skill to upload the image to the project.\r\nThis creates a new screen with the image as its content.\r\n\r\n#### 3. Refine with Edit\r\n\r\nOnce uploaded, use `list_screens` to find the newly created `screenId`, then\r\ncall `edit_screens` with a descriptive prompt to refine the design:\r\n\r\n```json\r\n{\r\n  \"projectId\": \"...\",\r\n  \"selectedScreenIds\": [\"<uploaded-screen-id>\"],\r\n  \"prompt\": \"[Describe what to adjust, enhance, or recreate from this mockup]\"\r\n}\r\n```\r\n\r\n> [!TIP]\r\n> For best results, describe the intent behind the image rather than just saying\r\n> \"make it look like this\". For example: \"This is a dashboard mockup — recreate\r\n> it with a proper data table, sidebar navigation, and chart widgets.\"\r\n\r\n#### 4. Present AI Feedback\r\n\r\nAlways show the text description and suggestions from `outputComponents` to the\r\nuser.\r\n\r\n#### 5. Download Design Assets\r\n\r\nDownload the HTML and screenshot urls from `outputComponents` to the\r\n`.stitch/designs` directory.\r\n\r\n- **Naming**: Use the screen ID or a descriptive slug for the filename.\r\n- **Tools**: Use `curl -o` via `run_command` or similar.\r\n- **Directory**: Ensure `.stitch/designs` exists.\r\n\r\n---\r\n\r\n### Edit Flow (Modify Existing Screen)\r\n\r\n#### 1. Identify the Screen\r\n\r\nUse `list_screens` or `get_screen` to find the correct `projectId` and\r\n`screenId`.\r\n\r\n#### 2. Formulate the Edit Prompt\r\n\r\nApply the Prompt Enhancement Pipeline, focusing on specificity:\r\n\r\n- **Location**: \"Change the color of the [primary button] in the [hero\r\n  section]...\"\r\n- **Visuals**: \"...to a darker blue (#004080) and add a subtle shadow.\"\r\n- **Structure**: \"Add a secondary button next to the primary one with the text\r\n  'Learn More'.\"\r\n\r\n#### 3. Apply the Edit\r\n\r\nCall the `edit_screens` tool.\r\n\r\n```json\r\n{\r\n  \"projectId\": \"...\",\r\n  \"selectedScreenIds\": [\"...\"],\r\n  \"prompt\": \"[Your targeted edit prompt]\"\r\n}\r\n```\r\n\r\n#### 4. Present AI Feedback\r\n\r\nAlways show the text description and suggestions from `outputComponents` to the\r\nuser.\r\n\r\n#### 5. Download Design Assets\r\n\r\nAfter editing, download the updated HTML and screenshot urls from\r\n`outputComponents` to the `.stitch/designs` directory, overwriting previous\r\nversions to ensure the local files reflect the latest edits.\r\n\r\n- **Naming**: Use the screen ID or a descriptive slug for the filename.\r\n- **Tools**: Use `curl -o` via `run_command` or similar.\r\n- **Directory**: Ensure `.stitch/designs` exists.\r\n\r\n#### 6. Update Project Metadata\r\n\r\nAfter downloading assets, update `.stitch/metadata.json` to reflect any changes\r\n(e.g., updated screen titles or new screen IDs from the edit). The metadata\r\nfile tracks all screens, their device types, and design system info. See the\r\n**manage-design-system** skill's `examples/metadata.json` for the format.\r\n\r\n#### 7. Verify and Repeat\r\n\r\n- Check the output screen to see if the changes were applied correctly.\r\n- If more polish is needed, repeat the edit flow with a new specific prompt.\r\n\r\n---\r\n\r\n### Generate Variants Flow (Explore Variations)\r\n\r\nUse this flow when the user wants to explore alternative layouts, color schemes,\r\nor content variations of an existing screen.\r\n\r\n#### 1. Identify the Screen\r\n\r\nUse `list_screens` or `get_screen` to find the correct `projectId` and\r\n`screenId`.\r\n\r\n#### 2. Configure Variant Options\r\n\r\nCall the `generate_variants` tool with the appropriate options:\r\n\r\n```json\r\n{\r\n  \"projectId\": \"...\",\r\n  \"selectedScreenIds\": [\"...\"],\r\n  \"prompt\": \"[Describe the direction for variants]\",\r\n  \"variantOptions\": {\r\n    \"variantCount\": 3,\r\n    \"creativeRange\": \"EXPLORE\",\r\n    \"aspects\": [\"LAYOUT\", \"COLOR_SCHEME\"]\r\n  }\r\n}\r\n```\r\n\r\n**Variant Options:**\r\n- **`variantCount`**: 1–5 variants (default: 3)\r\n- **`creativeRange`**: `REFINE` (subtle), `EXPLORE` (balanced), or `REIMAGINE`\r\n  (radical)\r\n- **`aspects`**: Focus on specific dimensions — `LAYOUT`, `COLOR_SCHEME`,\r\n  `IMAGES`, `TEXT_FONT`, `TEXT_CONTENT`, or leave empty for all\r\n\r\n#### 3. Present AI Feedback\r\n\r\nAlways show the text description and suggestions from `outputComponents` to the\r\nuser.\r\n\r\n#### 4. Download Design Assets\r\n\r\nDownload the variant HTML and screenshot urls from `outputComponents` to the\r\n`.stitch/designs` directory.\r\n\r\n- **Naming**: Use the screen ID or a descriptive slug for the filename.\r\n- **Tools**: Use `curl -o` via `run_command` or similar.\r\n- **Directory**: Ensure `.stitch/designs` exists.\r\n\r\n--------------------------------------------------------------------------------\r\n\r\n## 💡 Tips\r\n\r\n- **Be structural**: Break the page down into header, hero, features, and\r\n  footer in your prompt.\r\n- **Content first**: Describe what each section contains (text, images, CTAs)\r\n  rather than how it looks.\r\n- **Iterative Polish**: Prefer editing for targeted adjustments over full\r\n  re-generation.\r\n- **No theme leakage**: Never put hex codes, font names, or color roles in a\r\n  generation prompt — the design system handles all visual styling.\r\n- **Specify interactions**: Mention hover states, animations, and click behavior\r\n  rather than visual styling.\r\n- **Keep edits focused**: One edit at a time is often better than a long list of\r\n  changes.\r\n- **Reference components**: Use professional terms like \"navigation bar\", \"hero\r\n  section\", \"footer\", \"card grid\".\r\n- **Precise colors in edits**: Use hex codes for exact color matching when\r\n  editing existing screens.\r\n\r\n## 📚 References\r\n\r\n- [Design Mappings](references/design-mappings.md) — UI/UX keywords and\r\n  atmosphere descriptors.\r\n- [Prompting Keywords](references/prompt-keywords.md) — Technical terms Stitch\r\n  understands best.\r\n- [Enhanced Prompt Example](examples/enhanced-prompt.md) — Before/after prompt\r\n  enhancement.\n\n=== 工作流模块：stitch-loop ===\n# Stitch Build Loop\r\n\r\nYou are an **autonomous frontend builder** participating in an iterative site-building loop. Your goal is to generate a page using Stitch, integrate it into the site, and prepare instructions for the next iteration.\r\n\r\n## Overview\r\n\r\nThe Build Loop pattern enables continuous, autonomous website development through a \"baton\" system. Each iteration:\r\n1. Reads the current task from a baton file (`.stitch/next-prompt.md`)\r\n2. Generates a page using Stitch MCP tools\r\n3. Integrates the page into the site structure\r\n4. Writes the next task to the baton file for the next iteration\r\n\r\n## Prerequisites\r\n\r\n**Required:**\r\n- Access to the Stitch MCP Server\r\n- A Stitch project (existing or will be created)\r\n- A `.stitch/DESIGN.md` file (generate one using the `design-md` skill if needed)\r\n- A `.stitch/SITE.md` file documenting the site vision and roadmap\r\n\r\n**Optional:**\r\n- Chrome DevTools MCP Server — enables visual verification of generated pages\r\n\r\n## The Baton System\r\n\r\nThe `.stitch/next-prompt.md` file acts as a relay baton between iterations:\r\n\r\n```markdown\r\n---\r\npage: about\r\n---\r\nA page describing how jules.top tracking works.\r\n\r\n**DESIGN SYSTEM (REQUIRED):**\r\n[Copy from .stitch/DESIGN.md Section 6]\r\n\r\n**Page Structure:**\r\n1. Header with navigation\r\n2. Explanation of tracking methodology\r\n3. Footer with links\r\n```\r\n\r\n**Critical rules:**\r\n- The `page` field in YAML frontmatter determines the output filename\r\n- The prompt content must include the design system block from `.stitch/DESIGN.md`\r\n- You MUST update this file before completing your work to continue the loop\r\n\r\n## Execution Protocol\r\n\r\n### Step 1: Read the Baton\r\n\r\nParse `.stitch/next-prompt.md` to extract:\r\n- **Page name** from the `page` frontmatter field\r\n- **Prompt content** from the markdown body\r\n\r\n### Step 2: Consult Context Files\r\n\r\nBefore generating, read these files:\r\n\r\n| File | Purpose |\r\n|------|---------|\r\n| `.stitch/SITE.md` | Site vision, **Stitch Project ID**, existing pages (sitemap), roadmap |\r\n| `.stitch/DESIGN.md` | Required visual style for Stitch prompts |\r\n\r\n**Important checks:**\r\n- Section 4 (Sitemap) — Do NOT recreate pages that already exist\r\n- Section 5 (Roadmap) — Pick tasks from here if backlog exists\r\n- Section 6 (Creative Freedom) — Ideas for new pages if roadmap is empty\r\n\r\n### Step 3: Generate with Stitch\r\n\r\nUse the Stitch MCP tools to generate the page:\r\n\r\n1. **Discover namespace**: Run `list_tools` to find the Stitch MCP prefix\r\n2. **Get or create project**: \r\n   - If `.stitch/metadata.json` exists, use the `projectId` from it\r\n   - Otherwise, call `[prefix]:create_project`, then call `[prefix]:get_project` to retrieve full project details, and save them to `.stitch/metadata.json` (see schema below)\r\n   - After generating each screen, call `[prefix]:get_project` again and update the `screens` map in `.stitch/metadata.json` with each screen's full metadata (id, sourceScreen, dimensions, canvas position)\r\n3. **Generate screen**: Call `[prefix]:generate_screen_from_text` with:\r\n   - `projectId`: The project ID\r\n   - `prompt`: The full prompt from the baton (including design system block)\r\n   - `deviceType`: `DESKTOP` (or as specified)\r\n4. **Retrieve assets**: Before downloading, check if `.stitch/designs/{page}.html` and `.stitch/designs/{page}.png` already exist:\r\n   - **If files exist**: Ask the user whether to refresh the designs from the Stitch project or reuse the existing local files. Only re-download if the user confirms.\r\n   - **If files do not exist**: Proceed with download:\r\n     - `htmlCode.downloadUrl` — Download and save as `.stitch/designs/{page}.html`\r\n      - `screenshot.downloadUrl` — Append `=w{width}` to the URL before downloading, where `{width}` is the `width` value from the screen metadata (Google CDN serves low-res thumbnails by default). Save as `.stitch/designs/{page}.png`\r\n\r\n### Step 4: Integrate into Site\r\n\r\n1. Move generated HTML from `.stitch/designs/{page}.html` to `site/public/{page}.html`\r\n2. Fix any asset paths to be relative to the public folder\r\n3. Update navigation:\r\n   - Find existing placeholder links (e.g., `href=\"#\"`) and wire them to the new page\r\n   - Add the new page to the global navigation if appropriate\r\n4. Ensure consistent headers/footers across all pages\r\n\r\n### Step 4.5: Visual Verification (Optional)\r\n\r\nIf the **Chrome DevTools MCP Server** is available, verify the generated page:\r\n\r\n1. **Check availability**: Run `list_tools` to see if `chrome*` tools are present\r\n2. **Start dev server**: Use Bash to start a local server (e.g., `npx serve site/public`)\r\n3. **Navigate to page**: Call `[chrome_prefix]:navigate` to open `http://localhost:3000/{page}.html`\r\n4. **Capture screenshot**: Call `[chrome_prefix]:screenshot` to capture the rendered page\r\n5. **Visual comparison**: Compare against the Stitch screenshot (`.stitch/designs/{page}.png`) for fidelity\r\n6. **Stop server**: Terminate the dev server process\r\n\r\n> **Note:** This step is optional. If Chrome DevTools MCP is not installed, skip to Step 5.\r\n\r\n### Step 5: Update Site Documentation\r\n\r\nModify `.stitch/SITE.md`:\r\n- Add the new page to Section 4 (Sitemap) with `[x]`\r\n- Remove any idea you consumed from Section 6 (Creative Freedom)\r\n- Update Section 5 (Roadmap) if you completed a backlog item\r\n\r\n### Step 6: Prepare the Next Baton (Critical)\r\n\r\n**You MUST update `.stitch/next-prompt.md` before completing.** This keeps the loop alive.\r\n\r\n1. **Decide the next page**: \r\n   - Check `.stitch/SITE.md` Section 5 (Roadmap) for pending items\r\n   - If empty, pick from Section 6 (Creative Freedom)\r\n   - Or invent something new that fits the site vision\r\n2. **Write the baton** with proper YAML frontmatter:\r\n\r\n```markdown\r\n---\r\npage: achievements\r\n---\r\nA competitive achievements page showing developer badges and milestones.\r\n\r\n**DESIGN SYSTEM (REQUIRED):**\r\n[Copy the entire design system block from .stitch/DESIGN.md]\r\n\r\n**Page Structure:**\r\n1. Header with title and navigation\r\n2. Badge grid showing unlocked/locked states\r\n3. Progress bars for milestone tracking\r\n```\r\n\r\n## File Structure Reference\r\n\r\n```\r\nproject/\r\n├── .stitch/\r\n│   ├── metadata.json   # Stitch project & screen IDs (persist this!)\r\n│   ├── DESIGN.md       # Visual design system (from design-md skill)\r\n│   ├── SITE.md         # Site vision, sitemap, roadmap\r\n│   ├── next-prompt.md  # The baton — current task\r\n│   └── designs/        # Staging area for Stitch output\r\n│       ├── {page}.html\r\n│       └── {page}.png\r\n└── site/public/        # Production pages\r\n    ├── index.html\r\n    └── {page}.html\r\n```\r\n\r\n### `.stitch/metadata.json` Schema\r\n\r\nThis file persists all Stitch identifiers so future iterations can reference them for edits or variants. Check all internal links use correct relative paths |\n\n=== 工作流模块：stitch::manage-design-system ===\n# Design-System\r\n\r\nCreate a \"source of truth\" for your project's design language to Nest `Text` for inline styling. |\r\n| `<img>` | `Image` | Use `source={{ uri }}` for remote images, `require()` for local assets. |\r\n| `<button>`, `<a>` | `Pressable` | Prefer `Pressable` over `TouchableOpacity`. Use `onPress` instead of `onClick`. |\r\n| `<input>` | `TextInput` | Map `placeholder`, `value`, `onChangeText`. |\r\n| `<scroll container>` | `ScrollView` | For short lists only. Use `FlatList` for long or dynamic lists. |\r\n| `<ul>`/`<ol>` with many items | `FlatList` | Requires `data`, `renderItem`, `keyExtractor`. |\r\n| `<section>` with grouped data | `SectionList` | For grouped data with headers. Use tab navigator for tab-based layouts. |\r\n| `<select>` | Third-party picker or custom modal | React Native has no built-in select. |\r\n| `<svg>` | `react-native-svg` | Convert SVG markup to `Svg`, `Path`, `Circle`, etc. |\r\n| Root wrapper | `SafeAreaView` | Wrap top-level screens to avoid notch/status bar overlap. |\r\n\r\n### Style mapping\r\nCSS and Tailwind classes do not work in React Native. Convert all styles to `StyleSheet.create()`:\r\n\r\n**Layout**: Flexbox is the default layout system. `flexDirection` defaults to `'column'` (not `'row'` like web CSS).\r\n- `display: flex` is implicit on every `View`.\r\n- `justify-content` maps to `justifyContent`.\r\n- `align-items` maps to `alignItems`.\r\n- `gap` maps to `gap` (React Native 0.71+). For older versions, use `marginBottom` on children.\r\n\r\n**Dimensions**: Use numbers (not strings). `width: 100` means 100 density-independent pixels.\r\n- Percentage strings are supported: `width: '100%'`.\r\n- For responsive sizing, use `useWindowDimensions()` from `react-native`.\r\n- There is no `vw`/`vh`. Calculate from `Dimensions.get('window')`.\r\n\r\n**Typography**: All text styles must be on `Text` components, never on `View`.\r\n- `font-size` maps to `fontSize` (number, not string).\r\n- `font-weight` maps to `fontWeight` (string: `'400'`, `'700'`, `'bold'`).\r\n- `line-height` maps to `lineHeight` (number).\r\n- `letter-spacing` maps to `letterSpacing`.\r\n- `text-transform` maps to `textTransform`.\r\n- `color` applies to `Text` only.\r\n\r\n**Borders and shadows**:\r\n- `border-radius` maps to `borderRadius`.\r\n- `box-shadow` does not exist. Use `elevation` (Android) and `shadowColor`/`shadowOffset`/`shadowOpacity`/`shadowRadius` (iOS).\r\n- Use `Platform.select()` to apply platform-specific shadow styles.\r\n\r\n**Unsupported CSS properties** (handle manually or skip):\r\n- `hover`, `transition`, `animation` (use `react-native-reanimated` for animations).\r\n- `position: fixed` (use `position: 'absolute'` with manual offset).\r\n- `overflow: auto` (use `ScrollView`).\r\n- `z-index` works but only between sibling views.\r\n- `opacity` works. `visibility: hidden` does not exist; use conditional rendering or `opacity: 0`.\r\n\r\n**Color extraction**: Extract the Tailwind config from the HTML `<head>`. Map color tokens to a `theme.ts` constants file instead of hardcoding hex values in StyleSheet.\r\n\r\n### Platform-specific code\r\nWhen the design requires different behavior on iOS and Android:\r\n\r\n```typescript\r\nimport { Platform } from 'react-native';\r\n\r\nconst styles = StyleSheet.create({\r\n  shadow: Platform.select({\r\n    ios: {\r\n      shadowColor: '#000',\r\n      shadowOffset: { width: 0, height: 2 },\r\n      shadowOpacity: 0.1,\r\n      shadowRadius: 4,\r\n    },\r\n    android: {\r\n      elevation: 4,\r\n    },\r\n  }),\r\n});\r\n```\r\n\r\n## Architectural rules\r\n\r\n* **Atomic Design**: Organize components as atoms (buttons, labels, icons), molecules (input groups, cards), and organisms (headers, lists, forms). Place them in `src/components/atoms/`, `src/components/molecules/`, `src/components/organisms/`.\r\n* **Logic isolation**: Move event handlers, API calls, and business logic into custom hooks in `src/hooks/`. Components should only handle rendering.\r\n* **Data decoupling**: Move all static text, image URLs, and lists into `src/data/mockData.ts`. Reference them in `StyleSheet.create()`.\r\n* **Navigation**: Use React Navigation for screen transitions. Define screen types with `NativeStackScreenProps` or `BottomTabScreenProps`.\r\n* **Accessibility**: Every interactive element must have `accessibilityLabel` and `accessibilityRole`. Images need `accessibilityLabel`. Use `accessibilityState` for toggles and checkboxes.\r\n* **Safe areas**: Wrap top-level screen components with `SafeAreaView` from `react-native-safe-area-context` (not the one from `react-native`).\r\n* **Project specific**: Focus on the target project's needs and constraints. Leave Google license headers out of the generated React Native components.\r\n\r\n## Execution steps\r\n\r\n1. **Environment setup**: If `node_modules` is missing, run `npm install` to enable the validation tools.\r\n2. **Theme layer**: Create `src/theme.ts` from the extracted Tailwind config.
